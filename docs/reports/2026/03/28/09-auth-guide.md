## 1. 認証アーキテクチャ概要

Colyseus の認証システムは、**サーバー側**（`@colyseus/auth`）と**クライアント側**（`@colyseus/sdk` の `Auth` クラス）の2層で構成されている。

### サーバー側アーキテクチャ

`@colyseus/auth` パッケージは以下の4つのモジュールで構成される:

| モジュール | ファイル | 役割 |
|-----------|---------|------|
| `auth` | `src/auth.ts` | Express Router ベースの認証エンドポイント群（ログイン・登録・匿名認証・パスワードリセット） |
| `JWT` | `src/JWT.ts` | JWT トークンの署名・検証・ミドルウェア（Express + better-call 両対応） |
| `Hash` | `src/Hash.ts` | パスワードハッシュ（scrypt / sha1） |
| `oauth` | `src/oauth.ts` | OAuth プロバイダーとの連携（`grant` ライブラリ経由） |

### 認証フロー全体像

```
クライアント (SDK Auth)                サーバー (@colyseus/auth)
  |                                     |
  |-- POST /auth/register ------------->|  メール+パスワード登録
  |-- POST /auth/login ---------------->|  メール+パスワードログイン
  |-- POST /auth/anonymous ------------>|  匿名サインイン
  |-- signInWithProvider() ------------>|  OAuth（ポップアップ経由）
  |                                     |
  |<--- { user, token } ---------------|  JWT トークン返却
  |                                     |
  |-- Room join (token in options) ---->|  Room.onAuth() でトークン検証
```

### エンドポイント一覧

デフォルトプレフィックスは `/auth`（`auth.prefix` で変更可能）。

| メソッド | パス | 説明 |
|---------|------|------|
| POST | `/auth/register` | メール+パスワードで新規登録 |
| POST | `/auth/login` | メール+パスワードでログイン |
| POST | `/auth/anonymous` | 匿名サインイン |
| GET | `/auth/userdata` | JWT トークンからユーザーデータ取得（要認証） |
| POST | `/auth/forgot-password` | パスワードリセットメール送信 |
| GET | `/auth/reset-password` | パスワードリセットフォーム表示 |
| POST | `/auth/reset-password` | パスワードリセット実行 |
| GET | `/auth/confirm-email` | メールアドレス確認 |
| GET | `/auth/provider/:providerId` | OAuth プロバイダーへリダイレクト |
| GET | `/auth/provider/:providerId/callback` | OAuth コールバック |

### クライアント側アーキテクチャ

`@colyseus/sdk` の `Auth` クラスは、上記サーバーエンドポイントに対応するメソッドを提供する。トークンは `localStorage`（Web）に自動保存され、リクエスト時に `Authorization: Bearer <token>` ヘッダーとして自動付与される。

## 2. JWT 認証の設定方法

### 基本設定

JWT の動作には `JWT_SECRET` 環境変数が必須。

```bash
# 環境変数で設定（推奨）
export JWT_SECRET="your-secure-random-secret-here"
```

または、コード内で直接設定:

```typescript
import { JWT } from "@colyseus/auth";

JWT.settings.secret = "your-secure-random-secret-here";
```

### JWT 設定項目

`JWT.settings` オブジェクトで以下を制御できる:

| 設定 | デフォルト | 説明 |
|------|----------|------|
| `secret` | `process.env.JWT_SECRET` | JWT 署名・検証に使用するシークレット |
| `verify.algorithms` | `['HS256']` | 使用するアルゴリズム。配列の先頭が署名に使われる |

### JWT の署名と検証

```typescript
import { JWT } from "@colyseus/auth";

// 署名
const token = await JWT.sign({ userId: 123, email: "user@example.com" });

// 検証（成功時はデコードされたペイロードを返す）
const payload = await JWT.verify(token);

// オプション付き署名（有効期限など）
const tokenWithExpiry = await JWT.sign(
  { email: "user@example.com" },
  { expiresIn: "30m" }
);

// デコードのみ（署名検証なし）
const decoded = JWT.decode(token);
```

### JWT ミドルウェア

`auth.middleware()` は Express と better-call の両方で使えるデュアルミドルウェアを返す。

```typescript
import { auth } from "@colyseus/auth";

// Express ルートの保護
app.get("/protected", auth.middleware(), (req, res) => {
  // req.auth にデコード済み JWT ペイロードが入る
  res.json({ user: req.auth });
});
```

### サーバーへの認証ルート登録

```typescript
import { defineServer } from "@colyseus/core";
import { auth } from "@colyseus/auth";

const server = defineServer({
  // ... transport, rooms の設定
  express: (app) => {
    app.use(auth.prefix, auth.routes({
      onFindUserByEmail: async (email) => {
        // DB からユーザーを検索して返す（password フィールド必須）
        return await db.users.findOne({ email });
      },
      onRegisterWithEmailAndPassword: async (email, hashedPassword, options) => {
        // DB にユーザーを保存
        return await db.users.create({ email, password: hashedPassword });
      },
      onRegisterAnonymously: async (options) => {
        // 匿名ユーザーの作成（オプション）
        return await db.users.create({ anonymous: true, ...options });
      },
    }));
  }
});
```

### AuthSettings コールバック一覧

| コールバック | 必須 | 説明 |
|------------|------|------|
| `onFindUserByEmail` | 必須 | メールアドレスからユーザーを検索。`password` フィールドを含むオブジェクトを返す |
| `onRegisterWithEmailAndPassword` | 必須 | 新規ユーザー登録。パスワードはハッシュ済みで渡される |
| `onRegisterAnonymously` | 任意 | 匿名ユーザー登録。未設定時は自動生成 ID が返される |
| `onParseToken` | 任意 | JWT デコード後のカスタム変換。デフォルトはペイロードをそのまま返す |
| `onGenerateToken` | 任意 | トークン生成のカスタマイズ。デフォルトはユーザーデータ全体を JWT に埋め込む |
| `onHashPassword` | 任意 | パスワードハッシュのカスタマイズ。デフォルトは scrypt + `AUTH_SALT` |
| `onSendEmailConfirmation` | 任意 | メールアドレス確認メール送信 |
| `onEmailConfirmed` | 任意 | メールアドレス確認完了時の処理 |
| `onForgotPassword` | 任意 | パスワードリセットメール送信 |
| `onResetPassword` | 任意 | パスワードリセット実行 |
| `onOAuthProviderCallback` | 任意 | OAuth 認証成功時のコールバック |

### パスワードハッシュ

`Hash` クラスは2つのアルゴリズムを提供する:

| アルゴリズム | 説明 |
|------------|------|
| `scrypt` (デフォルト) | Node.js `crypto.scrypt` を使用。64バイトの派生キーを hex で出力 |
| `sha1` | SHA1 + ソルト。レガシー用途向け |

ソルトは `process.env.AUTH_SALT` から取得される（未設定時は `"## SALT ##"`）。

```typescript
import { Hash } from "@colyseus/auth";

// アルゴリズム変更
Hash.algorithm = "scrypt"; // デフォルト

// カスタムアルゴリズムの追加
Hash.algorithms["bcrypt"] = async (password, salt) => {
  // カスタム実装
};
```

## 3. OAuth プロバイダー設定

### 概要

Colyseus の OAuth は [`grant`](https://github.com/simov/grant) ライブラリをベースにしており、200以上のプロバイダーに対応している。セッション管理には `express-session` を使用し、Redis がある場合は自動的に `connect-redis` を使用する。

### プロバイダーの追加

```typescript
import { auth } from "@colyseus/auth";

// Google
auth.oauth.addProvider("google", {
  key: process.env.GOOGLE_CLIENT_ID,
  secret: process.env.GOOGLE_CLIENT_SECRET,
  scope: ["openid", "email", "profile"],
});

// GitHub
auth.oauth.addProvider("github", {
  key: process.env.GITHUB_CLIENT_ID,
  secret: process.env.GITHUB_CLIENT_SECRET,
  scope: ["read:user", "user:email"],
});

// Discord
auth.oauth.addProvider("discord", {
  key: process.env.DISCORD_CLIENT_ID,
  secret: process.env.DISCORD_CLIENT_SECRET,
  scope: ["identify", "email"],
});
```

### OAuthProviderConfig の設定項目

| 項目 | 型 | 説明 |
|------|-----|------|
| `key` | `string` | OAuth アプリの `client_id` または `consumer_key` |
| `secret` | `string` | OAuth アプリの `client_secret` または `consumer_secret` |
| `scope` | `string[]` | リクエストするスコープ |
| `nonce` | `boolean` | ランダム nonce 生成（OpenID Connect 用） |
| `custom_params` | `any` | カスタム認可パラメータ |
| `callback` | `string` | カスタムコールバック URL |
| `response` | `Array` | レスポンスタイプ（`tokens`, `raw`, `jwt`, `profile`） |

### OAuth コールバックの処理

```typescript
auth.settings.onOAuthProviderCallback = async (data, provider) => {
  // data.profile: プロバイダーから取得したプロフィール情報
  // data.access_token: アクセストークン
  // data.refresh_token: リフレッシュトークン（あれば）
  // data.upgradingToken: 匿名ユーザーのアップグレード時のみ存在
  // provider: "google" | "github" | "discord" など

  let user = await db.users.findOne({ [`${provider}Id`]: data.profile.sub });

  if (!user) {
    user = await db.users.create({
      [`${provider}Id`]: data.profile.sub,
      email: data.profile.email,
      name: data.profile.name,
    });
  }

  return user; // このオブジェクトが JWT に埋め込まれる
};
```

### OAuth フロー詳細

1. クライアントが `signInWithProvider("google")` を呼ぶ
2. ポップアップウィンドウが開き、`/auth/provider/google` にアクセス
3. `grant` ミドルウェアが Google の認可画面にリダイレクト
4. ユーザーが認可するとコールバック URL にリダイレクト
5. `/auth/provider/google/callback` で `onOAuthProviderCallback` が呼ばれる
6. JWT トークンを生成し、`window.postMessage()` でポップアップからメインウィンドウへ結果を送信
7. ポップアップが閉じ、クライアント側で `AuthResponse` を受け取る

### 匿名ユーザーからのアップグレード

既存の匿名トークンを OAuth 登録時に引き継ぐことができる:

```typescript
// クライアント側: 既存トークンがある場合、自動的に送信される
const result = await client.auth.signInWithProvider("google");
// サーバー側: data.upgradingToken に匿名ユーザーの JWT ペイロードが入る
```

### 必要な環境変数

```bash
SESSION_SECRET="your-session-secret"  # express-session 用
```

### プロフィールデータの変換

`oauth.transformProfileData()` は、プロバイダーごとのレスポンス形式の差異を吸収する。例えば Twitch は `{ data: [profile] }` 形式で返すが、自動的に最初の要素が取り出される。

## 4. BetterAuth 統合の概要

### better-call とは

`@colyseus/better-call` は Colyseus リポジトリ内の git submodule（`packages/better-call/`）として管理されている、型安全な API エンドポイント定義フレームワークである。Express の代替として使用でき、ミドルウェアチェーンとコンテキスト型推論を提供する。

### JWT ミドルウェアの better-call 対応

`JWT.middleware()` は Express と better-call の両方で動作するデュアルミドルウェアとして実装されている。内部的には引数の数で Express / better-call を判別する:

- **引数が3つの場合**: Express ミドルウェア `(req, res, next)` として動作。`express-jwt` ライブラリを使用し、`req.auth` にデコード済みペイロードを設定する
- **引数が1つの場合**: better-call ミドルウェア `(ctx)` として動作。`Authorization` ヘッダーから手動でトークンを抽出・検証し、`ctx.context.auth` にデコード済みペイロードを設定する

```typescript
import { createEndpoint } from "@colyseus/better-call";
import { auth } from "@colyseus/auth";

// better-call でのエンドポイント定義例
const protectedRoute = createEndpoint("/protected-route", {
  method: "GET",
  use: [auth.middleware()],
}, async (ctx) => {
  // ctx.context.auth にデコード済み JWT ペイロードが入る
  return { user: ctx.context.auth };
});
```

### 型定義

```typescript
// JWT.ts で定義されている型
export type JWTAuthMiddleware<T = JwtPayload> =
  ((req: Request, res: Response, next: NextFunction) => void) &  // Express
  { options: MiddlewareOptions } &                                 // better-call メタ
  ((ctx: MiddlewareInputContext<MiddlewareOptions>) => Promise<{ auth: T }>);  // better-call
```

better-call ミドルウェアの認証失敗時は `APIError(401)` がスローされる。

## 5. Room の onAuth メソッドでのカスタム認証

### 概要

Room レベルの認証は、クライアントが Room に join する際に実行される。`@colyseus/auth` とは独立しており、任意の認証ロジックを実装できる。

### 2つの onAuth パターン

Colyseus 0.17.x では、`onAuth` には**静的メソッド**と**インスタンスメソッド**の2パターンがある。静的メソッドが優先される。

#### パターン1: 静的 onAuth（推奨）

MatchMaker レベルで座席予約前に実行される。Room インスタンスが不要なため、認証失敗時のリソース消費が少ない。

```typescript
import { Room, type AuthContext } from "@colyseus/core";

class MyRoom extends Room {
  static async onAuth(token: string, options: any, context: AuthContext) {
    // token: クライアントから送信されたトークン
    // options: joinRoom 時のオプション
    // context: { token, headers, ip, req? }

    if (!token) {
      throw new Error("Authentication required");
    }

    const payload = await JWT.verify(token);
    const user = await db.users.findById(payload.userId);

    if (!user) {
      throw new Error("User not found");
    }

    // 返り値が client.auth に設定される
    return { userId: user.id, username: user.name };
  }
}
```

#### パターン2: インスタンス onAuth（非推奨予定）

Room インスタンス上で実行される。静的 `onAuth` が定義されている場合、インスタンス版は無視される。

```typescript
class MyRoom extends Room {
  onAuth(client: Client, options: any, context: AuthContext) {
    // client: 接続中のクライアントオブジェクト
    // options: joinRoom 時のオプション
    // context: { token, headers, ip }

    // false または falsy を返すと AUTH_FAILED エラー
    return true;
  }
}
```

### AuthContext の型定義

```typescript
type AuthContext = {
  token?: string;       // クライアントから送信されたトークン
  headers: Headers;     // HTTP ヘッダー
  ip: string | string[];  // クライアント IP アドレス
  req?: any;            // WebSocketTransport の場合のみ、生の HTTP リクエスト
};
```

### 認証フローの実行タイミング

1. **静的 onAuth**: `matchMaker.joinOrCreate()` / `matchMaker.join()` / `matchMaker.create()` 時に `callOnAuth()` 内部関数で実行される。座席予約（`reserveSeatFor`）の**前**に実行され、認証データが座席予約と共に Room プロセスに送られる
2. **インスタンス onAuth**: クライアントが実際に WebSocket 接続した後、`_onJoin()` 内で実行される

### defineRoom での使用

`defineRoom` API を使う場合も同様に `onAuth` を定義できる:

```typescript
import { defineRoom, defineServer } from "@colyseus/core";

const server = defineServer({
  rooms: {
    game: defineRoom({
      onAuth: async (client, options, context) => {
        // カスタム認証ロジック
        return true;
      },
      onJoin: (client, options, auth) => {
        // auth には onAuth の返り値が入る（非推奨: client.auth を使用）
        console.log(client.auth);
      },
    }),
  },
});
```

### 座席予約タイムアウト

`onAuth` の処理に時間がかかる場合、`seatReservationTime` を調整する:

```typescript
class MyRoom extends Room {
  seatReservationTime = 30; // デフォルト 15秒
}
```

## 6. クライアント側の認証フロー

### Auth クラスの概要

`@colyseus/sdk` の `Auth` クラスは、サーバーの `@colyseus/auth` エンドポイントに対応するクライアント側の API を提供する。

### 初期化

```typescript
import { Client } from "colyseus";

const client = new Client("ws://localhost:2567");
// client.auth は自動的に Auth インスタンスとして初期化される
```

### 認証メソッド

#### メール+パスワード登録

```typescript
const { user, token } = await client.auth.registerWithEmailAndPassword(
  "user@example.com",
  "password123",
  { displayName: "Player1" }  // オプション（onRegisterWithEmailAndPassword の第3引数に渡される）
);
```

#### メール+パスワードログイン

```typescript
const { user, token } = await client.auth.signInWithEmailAndPassword(
  "user@example.com",
  "password123"
);
```

#### 匿名サインイン

```typescript
const { user, token } = await client.auth.signInAnonymously({
  // オプションデータ
  displayName: "Guest"
});
```

#### OAuth プロバイダーでサインイン

```typescript
const { user, token } = await client.auth.signInWithProvider("google", {
  prefix: "/auth/provider",  // カスタムプレフィックス（オプション）
  width: 480,   // ポップアップ幅（デフォルト: 480）
  height: 768,  // ポップアップ高さ（デフォルト: 768）
});
```

OAuth は `window.open()` でポップアップを開き、`window.postMessage()` で結果を受け取る。ユーザーがポップアップを閉じた場合は `"cancelled"` で reject される。

#### パスワードリセットメール送信

```typescript
await client.auth.sendPasswordResetEmail("user@example.com");
```

#### サインアウト

```typescript
await client.auth.signOut();
// トークンが localStorage から削除され、onChange リスナーに null が通知される
```

### トークン管理

```typescript
// 現在のトークンを取得
const token = client.auth.token;

// トークンは自動的に localStorage に保存される
// キー名: "colyseus-auth-token"（auth.settings.key で変更可能）
```

### 認証状態の監視

```typescript
const unbind = client.auth.onChange((response) => {
  if (response.user) {
    console.log("ログイン中:", response.user);
  } else {
    console.log("未ログイン");
  }
});

// 監視解除
unbind();
```

`onChange` の初回呼び出し時、保存済みトークンがあれば自動的に `/auth/userdata` を呼び出してユーザーデータを取得する。トークンが無効な場合は `{ user: null, token: undefined }` が通知される。

### ユーザーデータの取得

```typescript
const { user } = await client.auth.getUserData();
```

### AuthResponse 型

```typescript
interface AuthResponse<UserData = any> {
  user: UserData;
  token: string;
}
```

## 7. セキュリティ考慮事項

### JWT シークレット

- **`JWT_SECRET` は必ず本番環境で強力なランダム文字列を設定すること**。未設定の場合、コンソールにエラーが出力されるが、サーバーは起動してしまう
- デフォルトアルゴリズムは `HS256`。必要に応じて `JWT.settings.verify.algorithms` で変更可能

### パスワードハッシュ

- デフォルトのハッシュアルゴリズムは `scrypt`（Node.js 標準の `crypto.scrypt`）
- **`AUTH_SALT` 環境変数を必ず設定すること**。未設定時のデフォルトソルトは `"## SALT ##"` であり、本番環境では危険
- `onHashPassword` コールバックで bcrypt 等の別アルゴリズムに変更可能

### パスワードポリシー

- 現在の最低パスワード長は **6文字**（`isValidPassword` 関数でハードコード）
- より厳しいポリシーが必要な場合は、`onRegisterWithEmailAndPassword` 内で独自バリデーションを追加する

### メールバリデーション

- サーバー側で正規表現 `/^[^\s@]+@[^\s@]+\.[a-z]{2,}$/i` によるバリデーションが行われる

### OAuth セッション

- `SESSION_SECRET` 環境変数が必要。OAuth セッション管理に `express-session` を使用
- Redis が利用可能な場合（MatchMaker の Presence が Redis の場合）、自動的に `connect-redis` がセッションストアとして使われる。それ以外の場合はメモリストアが使用される

### パスワードリセットトークン

- リセットトークンの有効期限は **30分**（`RESET_PASSWORD_TOKEN_EXPIRATION_MINUTES`）
- 使用済みトークンは Redis/Presence の `setex` で無効化され、同じトークンの再利用を防止する

### OAuth コールバックの origin

- OAuth コールバックページは `window.postMessage(response, '*')` を使用してメインウィンドウに結果を送信している
- ソースコード内のコメントにも記載があるが、origin が `'*'` であるため、理論的にはメッセージの傍受が可能。OAuth プロバイダー側でトークンと origin が既に検証されているという前提に基づいている
- クライアント側の `signInWithProvider` でも `event.origin` のチェックが TODO として残されている

### バックエンド URL の自動検出

- `auth.backend_url` が未設定の場合、最初のリクエストの `Host` ヘッダーから自動検出される
- **Host ヘッダーインジェクション攻撃を軽減するため、検出は1回限り**で行われ、以降はミドルウェアスタックから削除される
- 本番環境では `auth.backend_url` を明示的に設定することを推奨する

### JWT ペイロードからのパスワード除去

- ログイン・登録時、`onFindUserByEmail` から返されたユーザーオブジェクトから `password` フィールドが `delete` で削除された後に JWT へ埋め込まれる
- `onGenerateToken` をカスタマイズする場合は、パスワードハッシュが含まれないよう注意すること
