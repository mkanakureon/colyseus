### RoomInternalState 遷移

| 状態 | 値 | 説明 |
|------|-----|------|
| `CREATING` | 0 | `__init()` 実行中〜`onCreate()` 完了前。`metadata` や `roomId` のセットが許可される |
| `CREATED` | 1 | `onCreate()` 完了後。通常運用状態。`driver.persist()` が有効になる |
| `DISPOSING` | 2 | `disconnect()` または `#_disposeIfEmpty()` により破棄処理中 |

## 2. 各フェーズの詳細

### 2.1 onCreate(options)

- **呼び出し元**: `MatchMaker.handleCreateRoom()`
- **タイミング**: `room.__init()` の後、`_internalState` が `CREATING` → `CREATED` に切り替わる前
- **引数**: クライアントオプションとハンドラーのデフォルトオプションをマージしたもの（`merge({}, clientOptions, handler.options)`）
- **例外時**: `ServerError` としてラップされ、Room は生成されない
- **注意点**:
  - `this.metadata = ...` による直接代入は `onCreate()` 内でのみ許可される（`CREATING` 状態でのみ可）
  - `this.roomId` の上書きも `onCreate()` 内でのみ可能
  - `this.state = new MyState()` で Serializer が自動設定される（Schema 検出）

### 2.2 onAuth(client, options, context)

- **2つの形式がある**:
  1. **static onAuth(token, options, context)** — クラスメソッド。MatchMaker の `callOnAuth()` で呼ばれ、座席予約**前**に実行される
  2. **instance onAuth(client, options, context)** — インスタンスメソッド。`_onJoin()` 内で座席予約**後**に実行される（非推奨化の方向）
- **static が定義されている場合、instance は無視される**
- **デフォルト実装**: `return true`（認証なし）
- **falsy を返した場合**: `AUTH_FAILED` エラーがスローされる
- **例外時**: 座席予約が解除され（`delete _reservedSeats[sessionId]`）、クライアント数がデクリメントされる

### 2.3 onJoin(client, options, auth)

- **呼び出し元**: `_onJoin()` 内、`onAuth()` 成功後
- **引数**:
  - `client`: 接続済みクライアント（既に `clients` 配列に追加済み）
  - `options`: 座席予約時の `joinOptions`
  - `auth`: `client.auth` の値（`onAuth` の戻り値）
- **onJoin 中にクライアントが切断した場合**: `client.state` が `LEAVING` になり、`_onLeave` が呼ばれてエラーがスローされる
- **成功後**: `_events.emit('join')` → MatchMaker の `onClientJoinRoom()` → CCU インクリメント

### 2.4 onDrop(client, code)

- **呼び出し元**: `_onLeave()` 内、`code !== CloseCode.CONSENTED` かつ `client.state !== RECONNECTING` の場合
- **目的**: 意図しない切断（ネットワーク障害、サーバーシャットダウン等）への対応
- **`onDrop` が未定義の場合**: `onLeave` がフォールバックとして呼ばれる
- **重要**: `allowReconnection()` は `onDrop` 内で呼ぶべきメソッド
- **`onDrop` 後の追加処理**:
  - `allowReconnection()` が呼ばれた場合 → 再接続待機状態に入る
  - 再接続がタイムアウトまたは拒否された場合 → `onLeave()` が追加で呼ばれる（`#_onAfterLeave` 内）

### 2.5 onReconnect(client)

- **呼び出し元**: `_onJoin()` 内、再接続フロー（`isWaitingReconnection === true`）の場合
- **タイミング**: `allowReconnection()` の Deferred が resolve された後
- **引数**: 新しい `client`（`auth`、`userData`、`view` は前のクライアントから復元済み）
- **例外時**: `_onLeave(client, FAILED_TO_RECONNECT)` が呼ばれ、再接続失敗となる
- **成功後**: `client.state` が `RECONNECTING` → `JOINING` に遷移

### 2.6 onLeave(client, code)

- **呼び出し元**: `_onLeave()` 内
- **呼ばれるケース**:
  1. `code === CloseCode.CONSENTED`（自発的退出）の場合に直接呼ばれる
  2. `onDrop` が未定義で非自発的切断の場合にフォールバックとして呼ばれる
  3. `onDrop` + `allowReconnection()` 後、再接続が失敗/タイムアウトした場合に `#_onAfterLeave` 内で呼ばれる
- **close code 一覧**:
  - `CloseCode.CONSENTED` (4000): 自発的退出
  - `CloseCode.WITH_ERROR` (4002): エラーによる切断
  - `CloseCode.SERVER_SHUTDOWN` (4003): サーバーシャットダウン
  - `CloseCode.MAY_TRY_RECONNECT` (4004): 再接続可能（devMode）
  - `CloseCode.FAILED_TO_RECONNECT` (4005): 再接続失敗
- **並行性**: `#_onLeaveConcurrent` カウンターで同時実行数を追跡。`#_disposeIfEmpty()` は全 onLeave が完了するまで dispose しない

### 2.7 onDispose()

- **呼び出し元**: `#_dispose()` 内
- **タイミング**: `_internalState` が `DISPOSING` に設定された後、`driver.remove()` の後
- **後処理**:
  - パッチインターバル停止（`clearInterval`）
  - シミュレーションインターバル停止
  - autoDispose タイムアウト停止
  - `clock.clear()` + `clock.stop()` — 全タイマー/インターバル停止
- **MatchMaker 側**: `disposeRoom()` で `stats.local.roomCount--`、IPC チャンネル解除

## 3. 座席予約（Seat Reservation）の仕組み

### 概要

座席予約は、マッチメイキングと実際の WebSocket 接続の間のギャップを埋める仕組み。クライアントが `joinOrCreate()` 等を呼ぶと、まず座席が予約され、`ISeatReservation`（`sessionId` + `roomId`）が返される。クライアントはこの情報を使って WebSocket 接続を確立する。

### データ構造

```typescript
// _reservedSeats[sessionId] = [joinOptions, authData, isConsumed, isWaitingReconnection]
private _reservedSeats: { [sessionId: string]: [any, any, boolean?, boolean?] } = {};
private _reservedSeatTimeouts: { [sessionId: string]: NodeJS.Timeout } = {};
```

### フロー

1. **MatchMaker.reserveSeatFor()** → `remoteRoomCall(roomId, '_reserveSeat', [sessionId, options, authData])`
2. **Room._reserveSeat()** 内:
   - `hasReachedMaxClients()` チェック（再接続の場合はスキップ）
   - `_reservedSeats[sessionId]` にエントリ追加
   - `#_incrementClientCount()` → listing の `clients` 数を更新、必要なら自動ロック
   - タイムアウト設定（デフォルト **15秒**、`seatReservationTimeout` で変更可能、環境変数 `COLYSEUS_SEAT_RESERVATION_TIME` でグローバル設定可能）
3. **タイムアウト時**: 予約エントリ削除 + `#_decrementClientCount()`
4. **クライアント接続時（_onJoin）**:
   - `isConsumed` フラグを `true` に設定
   - `onAuth` → `onJoin` 完了後、`_reservedSeats` から非 enumerable に変更（`hasReachedMaxClients` のカウント対象外にする）
   - 最終的に `delete _reservedSeats[sessionId]`

### maxClients との関係

```typescript
public hasReachedMaxClients(): boolean {
  return (
    (this.clients.length + Object.keys(this._reservedSeats).length) >= this.#_maxClients ||
    this._internalState === RoomInternalState.DISPOSING
  );
}
```

- `clients.length`（接続済み）+ `Object.keys(_reservedSeats).length`（予約中、enumerable のみ）で判定
- `onJoin` 成功後の予約は非 enumerable になるため、二重カウントが回避される

### ISeatReservation の構造

```typescript
{
  name: string;       // ルーム名
  sessionId: string;  // セッションID
  roomId: string;     // ルームID
  processId: string;  // プロセスID
  publicAddress?: string;  // パブリックアドレス（マルチプロセス時）
  devMode?: boolean;  // devMode フラグ
}
```

## 4. 再接続フロー

### 全体の流れ

```
クライアント切断（意図しない）
        │
        ▼
_onLeave(client, code)
  └─ code !== CONSENTED → onDrop(client, code)
        │
        ▼
onDrop 内で allowReconnection(client, seconds) 呼び出し
        │
        ▼
座席再予約（_reserveSeat with allowReconnection=true）
  + _reconnections[reconnectionToken] に Deferred を登録
        │
        ▼ （待機中...）
        │
クライアントが matchMaker.reconnect(roomId, { reconnectionToken }) を呼ぶ
        │
        ▼
MatchMaker.reconnect()
  └─ remoteRoomCall('checkReconnectionToken', [token])
     └─ sessionId を返す → buildSeatReservation()
        │
        ▼
クライアントが新しい WebSocket で接続
        │
        ▼
_onJoin() — isWaitingReconnection=true のパス
  ├─ clients.push(newClient)
  ├─ _reconnections[token].resolve(newClient) — Deferred 解決
  │   └─ auth/userData/view を前のクライアントから復元
  │   └─ client.state = RECONNECTING
  ├─ onReconnect(newClient)
  └─ client.state = JOINING → JOIN_ROOM 送信
```

### allowReconnection(client, seconds | "manual")

- **引数**:
  - `client`: 切断されたクライアント
  - `seconds`: タイムアウト秒数。`"manual"` の場合は `Infinity`（手動管理）
- **戻り値**: `Deferred<Client>` — Promise ライクなオブジェクト。`.reject()` で再接続を拒否できる
- **制約**:
  - `_enqueuedMessages !== undefined`（= onJoin が完了していない）クライアントには使用不可
  - `DISPOSING` 状態のルームでは使用不可
  - 同じ `reconnectionToken` に対する重複呼び出しは無視される（既存の Deferred を返す）
- **タイムアウト時**: `reconnection.reject(false)` → `onLeave()` が呼ばれる
- **成功時（resolve）**: 新クライアントに `auth`、`userData`、`view` が復元される

### checkReconnectionToken(token)

MatchMaker から呼ばれるメソッド。以下の順で確認する:

1. `_reconnections[token]` が存在し、対応する `_reservedSeats[sessionId]` の `isWaitingReconnection` フラグが `true` → sessionId を返す
2. `clients` 配列内に同じ `reconnectionToken` を持つクライアントが存在（古い接続がまだ生きている） → 古い接続を強制切断し sessionId を返す
3. いずれにも該当しない → `undefined` を返す

### ネットワーク切り替え時の特殊処理

Wi-Fi 切り替え等で古い接続がまだ生きている場合:
1. 新接続が `_onJoin()` に到達
2. `_reservedSeats[sessionId]` が存在しないが、`clients` に同じ sessionId のクライアントがいる
3. `_reconnectionAttempts[token]` に Deferred を作成して待機
4. 古い接続の `_onLeave` → `onDrop` → `allowReconnection()` が呼ばれると `_reconnectionAttempts[token].resolve()` される
5. 待機が解除され、通常の再接続フローに入る

## 5. Graceful Shutdown フロー

### 全体の流れ

```
gracefullyShutdown()
        │
        ▼
state = SHUTTING_DOWN
        │
        ▼ (devMode の場合)
cacheRoomHistory(rooms) — ルーム状態をキャッシュ
        │
        ▼
lockAndDisposeAll()
  ├─ stats.excludeProcess(processId) — 新規ルーム作成を停止
  ├─ 全ルームを lock() — 新規参加を停止
  ├─ 全ルームの onBeforeShutdown() 呼び出し
  │   └─ デフォルト: disconnect(CloseCode.SERVER_SHUTDOWN)
  │   └─ devMode: disconnect(CloseCode.MAY_TRY_RECONNECT)
  └─ noActiveRooms イベントを待機（全ルーム dispose 完了まで）
        │
        ▼
removeRoomsByProcessId(processId) — キャッシュからルーム情報を削除
        │
        ▼
presence.unsubscribe(processChannel) — IPC チャンネル解除
        │
        ▼
disconnectAll() — 残存クライアント全切断（安全策）
```

### onBeforeShutdown()

- **デフォルト実装**: `this.disconnect(isDevMode ? CloseCode.MAY_TRY_RECONNECT : CloseCode.SERVER_SHUTDOWN)`
- **カスタマイズ可能**: ユーザーが override して独自のシャットダウンロジックを実装できる
- **devMode の場合**: `lockAndDisposeAll()` はデフォルト実装を強制的に呼ぶ（`Room.prototype.onBeforeShutdown.call(room)`）

### devMode 固有の機能

- `onCacheRoom()`: シャットダウン時にルームデータをキャッシュするためのフック
- `onRestoreRoom(cached)`: 起動時にキャッシュからルームを復元するためのフック
- `cacheRoomHistory(rooms)`: 全ルームの情報を Presence に保存

## 6. ロック/アンロックの状態管理

### ロックの種類

| 種類 | フラグ | トリガー |
|------|--------|---------|
| **明示的ロック** | `_lockedExplicitly = true` | `room.lock()` または `setMatchmaking({ locked: true })` |
| **自動ロック（maxClients）** | `#_maxClientsReached = true` | `hasReachedMaxClients()` が `true` になった時 |

### ロック状態の管理

```typescript
#_locked: boolean = false;          // 実際のロック状態
_lockedExplicitly: boolean = false;  // ユーザーが明示的にロックしたか
#_maxClientsReached: boolean = false; // maxClients に達したか
```

### ロック/アンロックのルール

1. **`lock()`**: `#_locked = true`。引数なしで呼ぶと `_lockedExplicitly = true`
2. **`unlock()`**: `#_locked = false`。引数なしで呼ぶと `_lockedExplicitly = false`
3. **自動ロック**: `#_incrementClientCount()` 内で `hasReachedMaxClients()` が true になると自動ロック
4. **自動アンロック**: `#_decrementClientCount()` 内で `#_maxClientsReached` が true かつ `_lockedExplicitly` が false の場合にアンロック
5. **明示的ロック優先**: `_lockedExplicitly === true` の場合、クライアント減少による自動アンロックは発生しない

### Listing との同期

- `lock()` / `unlock()` は `matchMaker.driver.update()` で listing を永続化する
- `setMatchmaking()` 経由の場合は一括で persist される
- `#_incrementClientCount()` / `#_decrementClientCount()` は `$inc` + `$set` で listing を更新

## 7. パッチサイクル（状態同期タイミング）

### デフォルト設定

| 設定 | デフォルト値 | 説明 |
|------|-------------|------|
| `patchRate` | **50ms (20fps)** | 状態同期の送信間隔 |
| `setSimulationInterval` | **16.66ms (60fps)** | ゲームループの実行間隔 |

### パッチサイクルの流れ

```
setInterval(broadcastPatch, patchRate)  ← 50ms ごと
        │
        ▼
broadcastPatch()
  ├─ onBeforePatch(state)     ← ユーザー定義フック（任意）
  ├─ clock.tick()             ← simulationInterval 未設定時のみ
  ├─ _serializer.applyPatches(clients, state)
  │   └─ 各クライアントに差分パッチを送信
  │   └─ StateView 対応: クライアントごとに異なるビューを送信可能
  └─ _dequeueAfterPatchMessages()
      └─ afterNextPatch: true で送られたメッセージをここで送信
```

### シミュレーションインターバルとの関係

```
setSimulationInterval(callback, 16.66ms)  ← 60fps ゲームループ
        │
        ▼
setInterval(() => {
  clock.tick();           ← タイマー更新
  callback(deltaTime);   ← ユーザー定義のゲームロジック
}, 16.66ms)
```

- `simulationInterval` と `patchRate` は独立したインターバル
- `simulationInterval` が設定されている場合、`broadcastPatch()` 内の `clock.tick()` はスキップされる（二重 tick 防止）
- `patchRate = 0` かつ `simulationInterval` なしの場合、`clock.tick()` のみのインターバルが設定される

### afterNextPatch メッセージ

- `client.send(type, message, { afterNextPatch: true })` で送信すると、次の `broadcastPatch()` 後にメッセージが送信される
- `this.broadcast(type, message, { afterNextPatch: true })` も同様
- キューは `_afterNextPatchQueue` に蓄積され、`_dequeueAfterPatchMessages()` で処理される

### クライアントへの初期状態送信

- `_onJoin()` 完了後、クライアントが `Protocol.JOIN_ROOM` を返すと `client.state = JOINED` に遷移
- その後 `sendFullState(client)` で現在の完全な状態が送信される
- `onJoin()` 中に `client.send()` したメッセージは `_enqueuedMessages` にキューイングされ、`JOINED` 後にフラッシュされる

### maxMessagesPerSecond

- クライアントが1秒間に送信できるメッセージ数の上限（デフォルト: `Infinity`）
- 超過した場合、クライアントは `CloseCode.WITH_ERROR` で強制切断される
- `clock.currentTime` ベースでカウントがリセットされる
