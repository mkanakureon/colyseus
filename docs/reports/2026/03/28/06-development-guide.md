# Colyseus Development Guide
> **Repository:** colyseus | **Generated by:** Claude Opus 4.6 | **Date:** 2026-03-28
> **Scope:** 全パッケージ

## 1. ローカル開発セットアップ手順

### 前提条件

| ツール | バージョン | 備考 |
|--------|-----------|------|
| Node.js | >= 20.x | CI では Node 22 を使用 |
| pnpm | >= 8.15.0 | CI では 10.9.0。`corepack enable` で有効化推奨 |
| Git | 最新 | サブモジュール対応が必要 |
| Redis | 6+ | ドライバ/プレゼンスのテストに必要 |

### セットアップ手順

```bash
# 1. リポジトリをクローン（サブモジュール含む）
git clone --recursive https://github.com/colyseus/colyseus.git
cd colyseus

# 2. 依存関係のインストール
#    preinstall フックで git submodule update --init --recursive が自動実行される
#    postinstall フックで @better-auth/root のビルドが自動実行される
pnpm install

# 3. 全パッケージをビルド
pnpm build
```

### Git サブモジュール

`packages/better-call` が Git サブモジュールとして管理されている（BetterAuth 統合用）。

```bash
# サブモジュールの手動初期化（pnpm install で自動実行されるため通常不要）
git submodule update --init --recursive

# サブモジュールを最新に更新
pnpm run update-submodule
```

`pnpm install` の `preinstall` フックでサブモジュールが自動初期化され、`postinstall` フックで `@better-auth/root` のビルドが実行されるため、通常は `pnpm install` だけで完結する。

### Redis のセットアップ

Redis ドライバ・プレゼンスのテストに Redis サーバーが必要。

```bash
# macOS (Homebrew)
brew install redis
brew services start redis

# Docker
docker run -d --name colyseus-redis -p 6379:6379 redis:6

# 動作確認
redis-cli ping   # => PONG
```

Redis が起動していない場合、`@colyseus/redis-driver` と `@colyseus/redis-presence` 関連のテストが失敗する。

## 2. ビルド手順

### ビルドシステムの概要

ルートの `build.ts`（`tsx build.ts` で実行）が esbuild を使って全パッケージを一括ビルドする。

- **出力形式:** CJS (`.cjs`) + ESM (`.mjs`) のデュアルフォーマット + TypeScript 型定義 (`.d.ts`)
- **出力先:** 各パッケージの `build/` ディレクトリ
- **カスタムビルド:** `package.json` に `"build"` スクリプトを持つパッケージ（monitor, playground, sdk）はデフォルトビルドからスキップされ、独自のツールチェーン（Vite, Rollup）を使用

### 全体ビルド

```bash
# 標準ビルド（esbuild 対象パッケージのみ）
pnpm build

# 全パッケージビルド（better-call → デフォルト → sdk → playground → monitor）
pnpm run build-all
```

### 個別パッケージのビルド

```bash
# scope 指定でビルド（build.ts の --scope オプション）
pnpm build -- --scope @colyseus/core

# SDK のビルド（Rollup ベース、独自ビルド）
cd packages/sdk && pnpm build

# SDK の型定義のみ更新
cd packages/sdk && npx tsc

# Monitor のビルド（Vite ベース）
cd packages/monitor && pnpm build

# Playground のビルド（Vite + Tailwind ベース）
cd packages/playground && pnpm build
```

### ウォッチモード

```bash
# ファイル変更を監視して自動再ビルド
pnpm run watch
```

`compile-packages.mjs --watch` が実行される。esbuild 対象パッケージのソース変更を検知して自動的にリビルドする。

### 重要な注意点

- **テスト実行前には必ず `pnpm build` を実行すること。** テストはビルド済み出力（`build/` 以下の `.cjs`/`.mjs`）に対して実行される。ソースを変更しただけではテストに反映されない。
- `@colyseus/sdk` を変更した場合は `cd packages/sdk && npx tsc` で型定義を更新すること。

## 3. テスト実行手順

### テストフレームワーク

| パッケージ | フレームワーク |
|-----------|-------------|
| ほとんどのパッケージ | **Mocha** |
| `@colyseus/sdk`, `@colyseus/shared-types` | **Vitest** |
| `@colyseus/testing` | **Jest** (ts-jest) |

### 全テスト実行

```bash
# 全パッケージのテストを実行（Lerna 経由）
pnpm test
```

### 個別パッケージのテスト

```bash
# メインバンドル（統合テストの大半がここにある）
cd bundles/colyseus && pnpm test

# 特定パッケージ（--filter 指定）
pnpm --filter @colyseus/auth test
pnpm --filter @colyseus/core test
pnpm --filter @colyseus/sdk test
```

### 特定テストケースの実行（grep）

```bash
# テスト名で絞り込み（bundles/colyseus ディレクトリから実行）
cd bundles/colyseus && pnpm test -- --grep 'should create a room'

# 部分一致で絞り込み
cd bundles/colyseus && pnpm test -- --grep 'MatchMaker'
```

### テスト環境変数

`bundles/colyseus` のテストスクリプトでは以下の環境変数が自動設定される：

| 環境変数 | 値 | 用途 |
|---------|-----|------|
| `COLYSEUS_SEAT_RESERVATION_TIME` | `0.3` | シート予約のタイムアウト（秒） |
| `COLYSEUS_PRESENCE_SHORT_TIMEOUT` | `300` | プレゼンスの短縮タイムアウト（ms） |

### テストファイルの所在

主要な統合テストは `bundles/colyseus/test/` に集約されている：

- `Integration.test.ts` — クライアント接続・メッセージング
- `MatchMaker.test.ts` — マッチメイキング
- `Room.test.ts`, `RoomIntegration.test.ts` — ルームのライフサイクル
- `RoomReconnection.test.ts` — 再接続処理
- `Auth.test.ts` — 認証
- `Driver.test.ts` — ドライバ
- `GracefulShutdown.test.ts` — グレースフルシャットダウン
- `Server.test.ts` — サーバー全般

## 4. Example アプリの起動方法

### デフォルトの起動

```bash
# リポジトリルートから（Lerna 経由）
pnpm example

# または直接
cd packages/example && pnpm start
```

これにより `tsx src/app.config.ts` が実行され、デフォルトポート **2567** でサーバーが起動する。

### Example アプリの構成

| ファイル | 説明 |
|---------|------|
| `src/app.config.ts` | メインエントリポイント。`@colyseus/tools` の `config()` + `listen()` でサーバーを構成 |
| `src/MyRoom.ts` | サンプルルーム。Player スキーマ、移動メッセージ、再接続処理を実装 |
| `src/defineServer.ts` | `defineServer` を使った別パターンの構成例 |
| `src/ServerExpress.ts` | Express を使ったサーバー構成例 |
| `src/ServerH3.ts` | H3 トランスポートを使ったサーバー構成例 |
| `src/ServerRaw.ts` | 低レベル（Raw）サーバー構成例 |
| `src/ServerTCP.ts` | TCP トランスポートを使ったサーバー構成例 |

### H3 トランスポートで起動

```bash
cd packages/example && pnpm run example-h3
```

### 起動後のアクセス先

| URL | 内容 |
|-----|------|
| `http://localhost:2567/` | Playground（インタラクティブ API 操作画面） |
| `http://localhost:2567/monitor` | Monitor（ルーム監視ダッシュボード） |
| `http://localhost:2567/express` | Express エンドポイントの確認 |

### ポート番号

デフォルトは `2567`。環境変数 `PORT` で変更可能。PM2 クラスタモード時は `NODE_APP_INSTANCE` の値がポートに加算される。

### Redis/PostgreSQL との接続

`app.config.ts` 内の `options` でドライバ・プレゼンスを切り替えられる（デフォルトはコメントアウトされている）：

```typescript
options: {
  devMode: true,
  // driver: new RedisDriver(),
  // presence: new RedisPresence(),
  // driver: new PostgresDriver(),
},
```

有効化する場合は該当行のコメントを外し、Redis/PostgreSQL が起動していることを確認する。

## 5. CI パイプラインの構成

### ワークフロー概要

| 項目 | 値 |
|------|-----|
| ファイル | `.github/workflows/ci.yml` |
| 名前 | Tests |
| トリガー | `push` to `master`/`dev` ブランチ、全 `pull_request` |
| ランナー | `ubuntu-latest` |

### パイプラインステップ

```
1. Start Redis (v6)        — supercharge/redis-github-action@1.2.0
2. Checkout                — actions/checkout@v6 (submodules: recursive)
3. Install Node.js (v22)   — actions/setup-node@v4
4. Install pnpm (10.9.0)   — pnpm/action-setup@v4 (cache: true)
5. Install dependencies     — pnpm install
6. Build                    — pnpm build
7. Test                     — pnpm test
```

### 注意点

- Redis が必須サービスとして起動される（バージョン 6）
- サブモジュール（`packages/better-call`）は `checkout` 時に `recursive` で取得される
- ビルドとテストは逐次実行（ビルド出力に対してテストが走るため）
- pnpm のキャッシュが有効化されている

## 6. よくあるトラブルシューティング

### テストが失敗する：ビルドしていない

**症状:** テスト実行時に `Cannot find module` や古い挙動のまま。

**原因:** テストはビルド済み出力（`build/` ディレクトリ）に対して実行される。ソースコードを変更しただけでは反映されない。

**対処:**
```bash
pnpm build
pnpm test
```

### テストが失敗する：Redis に接続できない

**症状:** `ECONNREFUSED 127.0.0.1:6379` エラー。

**原因:** Redis サーバーが起動していない。

**対処:**
```bash
# Redis の起動確認
redis-cli ping

# 起動していなければ開始
brew services start redis
# または
docker run -d -p 6379:6379 redis:6
```

### サブモジュールが空

**症状:** `packages/better-call` が空ディレクトリ、`postinstall` でビルドエラー。

**原因:** サブモジュールが初期化されていない。

**対処:**
```bash
git submodule update --init --recursive
pnpm install
```

### SDK の型定義が古い

**症状:** `@colyseus/sdk` を変更したのに型エラーが残る。

**原因:** TypeScript 型定義 (`.d.ts`) が更新されていない。SDK は Rollup で独自ビルドするため、ルートの `pnpm build` では更新されない。

**対処:**
```bash
cd packages/sdk
npx tsc
```

### `pnpm build` で特定パッケージがスキップされる

**症状:** monitor, playground, sdk の変更がビルドに反映されない。

**原因:** これらのパッケージは `package.json` に独自の `"build"` スクリプトを持つため、ルートの `build.ts` からスキップされる。

**対処:**
```bash
# 全パッケージをビルドする場合
pnpm run build-all

# 個別にビルドする場合
cd packages/monitor && pnpm build
cd packages/playground && pnpm build
cd packages/sdk && pnpm build
```

### ESM/CJS 関連のインポートエラー

**症状:** `ERR_REQUIRE_ESM` や `Cannot use import statement outside a module`。

**原因:** このリポジトリは ESM-first（`"type": "module"`）。相対インポートには `.ts` 拡張子が必要。パッケージ間参照では `@source` カスタム条件が開発時のソース直接インポートに使われる。

**対処:**
- 相対インポートは `.ts` 拡張子付きで記述する
- `exports` フィールドの `@source` 条件を確認する
- CommonJS が必要な場合は `.cjs` 出力を参照する

### Mocha のテストタイムアウト

**症状:** テストが `timeout of 15000ms exceeded` で失敗。

**原因:** 統合テストはネットワーク通信や Redis 操作を含むため時間がかかることがある。

**対処:**
```bash
# タイムアウトを延長して実行
cd bundles/colyseus && pnpm test -- --timeout 30000
```
