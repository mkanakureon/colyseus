## 1. 単一サーバーから分散構成へのスケーリングパス

### 単一サーバー構成（デフォルト）

Colyseus はデフォルトで **LocalPresence** と **LocalDriver** を使用する。これらはプロセス内メモリにデータを保持するため、追加インフラなしで動作する。

```
packages/core/src/utils/Env.ts:
  getDefaultPresence() → new LocalPresence()
  getDefaultDriver()   → new LocalDriver()
```

**LocalPresence**（`packages/core/src/presence/LocalPresence.ts`）は EventEmitter ベースの Pub/Sub をプロセス内で提供する。**LocalDriver**（`packages/core/src/matchmaker/LocalDriver/LocalDriver.ts`）はルームキャッシュをメモリ配列で管理する。

この構成では：
- 1 プロセスのみで動作
- ルーム情報はプロセス内メモリに格納
- プロセス間通信は発生しない

### 分散構成への移行に必要な変更

分散構成に移行するには、以下の 3 つのコンポーネントを差し替える：

| コンポーネント | 単一サーバー | 分散構成 |
|---------------|-------------|---------|
| **Presence** | `LocalPresence` | `RedisPresence` |
| **Driver** | `LocalDriver` | `RedisDriver` |
| **ロードバランサー** | 不要 | Traefik（推奨） |

```typescript
// 分散構成の設定例
import { Server } from "@colyseus/core";
import { RedisPresence } from "@colyseus/redis-presence";
import { RedisDriver } from "@colyseus/redis-driver";

const server = new Server({
  presence: new RedisPresence("redis://localhost:6379"),
  driver: new RedisDriver("redis://localhost:6379"),
  publicAddress: "node-1.yourgamedomain.com",
});
```

### publicAddress の役割

各プロセスには一意な `publicAddress` を設定する。これは Traefik のルーティングルール生成に使用される。MatchMaker は `publicAddress` をルームキャッシュに含めてクライアントに返す。クライアントは受け取った `publicAddress` 宛てに WebSocket 接続を行う。

```
MatchMaker.ts (handleCreateRoom):
  if (publicAddress) {
    room['_listing'].publicAddress = publicAddress;
  }
```

## 2. Redis Driver + Redis Presence による分散マッチメイキング

### RedisDriver（`packages/drivers/redis-driver/src/RedisDriver.ts`）

**RedisDriver** は `MatchMakerDriver` インターフェースを実装し、ルームキャッシュを Redis のハッシュ（キー: `roomcaches`）に格納する。

主な特徴：

- **ルームデータの共有**: 全プロセスが同一の Redis ハッシュからルーム情報を読み書きする
- **並行クエリの最適化**: `_concurrentRoomCacheRequest` により、同時発生する `hgetall` を 1 回の Redis 呼び出しに集約する
- **ルーム名によるプレフィルタ**: JSON パース前に文字列マッチで名前フィルタを適用する（`"name":"${roomName}"` の存在チェック）
- **Redis Cluster 対応**: コンストラクタで `ClusterNode[]` を渡すと `ioredis.Cluster` を使用する
- **バッチクリーンアップ**: プロセス停止時、該当 processId のルームを 500 件単位で `hdel` する

```typescript
// Redis Cluster の使用例
const driver = new RedisDriver(
  [{ host: "redis-1", port: 6379 }, { host: "redis-2", port: 6379 }],
  { /* ClusterOptions */ }
);
```

### RedisPresence（`packages/presence/redis-presence/src/index.ts`）

**RedisPresence** は `Presence` インターフェースを実装し、Redis の Pub/Sub を使ったプロセス間メッセージングを提供する。

主な特徴：

- **2 つの Redis 接続**: `sub`（購読用）と `pub`（発行・データ操作用）を分離。Redis の制約上、`SUBSCRIBE` モード中のクライアントでは他のコマンドを実行できないため
- **JSON シリアライズ**: `publish` 時に `JSON.stringify`、受信時に `JSON.parse` を行う
- **EventEmitter によるローカル分配**: Redis からのメッセージを `subscriptions` EventEmitter 経由でコールバックに配信する
- **リスナー制限解除**: `this.sub.setMaxListeners(0)` により、多数のルーム・IPC チャネルを同時に購読可能
- **Redis Cluster 対応**: RedisDriver と同様に `ClusterNode[]` を渡せる

### マッチメイキングの分散フロー

```
Client → (Traefik) → Process A: joinOrCreate("battle")
  1. findOneRoomAvailable() → RedisDriver.findOne() → Redis HGETALL "roomcaches"
  2. 空きルームなし → createRoom()
  3. selectProcessIdToCreateRoom() → stats.fetchAll() → Redis HGETALL "roomcount"
     → 最もルーム数が少ないプロセスを選択（例: Process B）
  4. requestFromIPC(presence, processChannel(B), ...) → Redis PUBLISH
  5. Process B: subscribeIPC で受信 → handleCreateRoom() → ルーム生成
  6. Process B → Redis PUBLISH(ipc:{requestId}) → レスポンス返却
  7. Process A: reserveSeatFor() → クライアントに Process B の publicAddress を返却
  8. Client → Process B に WebSocket 接続
```

### 同時ルーム生成の排他制御

`joinOrCreate` では `concurrentJoinOrCreateRoomLock` により、同一フィルタ条件での重複ルーム生成を防止する。ロックが解放されると、待機中のリクエストは新しく作成されたルームの `roomId` を `presence.publish` 経由で受け取り、既存ルームに参加する。

## 3. IPC（プロセス間通信）の仕組み

### アーキテクチャ

IPC は `packages/core/src/IPC.ts` に実装されており、Redis Pub/Sub（Presence インターフェース）上に構築されたリクエスト/レスポンスパターンである。

### プロトコル

```
packages/core/src/Protocol.ts:
  IpcProtocol = { SUCCESS: 0, ERROR: 1, TIMEOUT: 2 }
```

### requestFromIPC（リクエスト側）

1. ユニークな `requestId` を生成する
2. レスポンス用チャネル `ipc:{requestId}` を購読する
3. 宛先チャネル（例: `process:{processId}`）に `[method, requestId, args]` を publish する
4. レスポンスを待ち、`[IpcProtocol.SUCCESS, data]` または `[IpcProtocol.ERROR, data]` を受信する
5. タイムアウト（デフォルト: `REMOTE_ROOM_SHORT_TIMEOUT`）時には `"ipc_timeout"` エラーを throw する

### subscribeIPC（応答側）

1. 指定チャネルを購読する
2. `[method, requestId, args]` を受信したら `replyCallback(method, args)` を実行する
3. 結果を `ipc:{requestId}` チャネルに `[SUCCESS, result]` として publish する
4. エラー時は `[ERROR, errorMessage]` を publish する

### IPC の使用箇所

| 用途 | チャネル | 内容 |
|------|---------|------|
| **リモートルーム生成** | `process:{processId}` | 他プロセスにルーム作成を依頼 |
| **リモートルーム操作** | `room:{roomId}` | `_reserveSeat`、`checkReconnectionToken` 等 |
| **ヘルスチェック** | `process:{processId}` | method: `"healthcheck"` → `true` を返す |

```
MatchMaker.ts (accept):
  subscribeIPC(presence, getProcessChannel(), (method, args) => {
    if (method === 'healthcheck') return true;
    else return handleCreateRoom.apply(undefined, args);
  });
```

### ヘルスチェック

起動時に `healthCheckAllProcesses()` が実行され、`roomcount` ハッシュに登録されている全 processId にヘルスチェック IPC を送信する。タイムアウトしたプロセスは `stats.excludeProcess()` で `roomcount` ハッシュから削除される。

また、各プロセスは 1 分間隔で `stats.persist()` を実行し、自身の存在を `roomcount` ハッシュに再登録する（`setAutoPersistInterval`）。

## 4. ロードバランサー設定（Traefik 統合）

### 概要

`@colyseus/traefik` パッケージ（`packages/traefik/src/index.ts`）は、各 Colyseus プロセスを Traefik に動的登録するユーティリティを提供する。

### 2 つのプロバイダーモード

#### Redis プロバイダー（推奨）

Traefik が直接 Redis からルーティング設定を読み取る。**Redis の Keyspace Notifications が必要。**

```yaml
# traefik-redis-provider.yml
providers:
  redis:
    rootKey: "traefik"
    endpoints:
      - "127.0.0.1:6379"
```

Colyseus は以下の Redis キーを設定する：

```
traefik/http/routers/{subdomain}/rule     → "Host(`node-1.yourgamedomain.com`)"
traefik/http/routers/{subdomain}/service  → "{subdomain}"
traefik/http/services/{subdomain}/loadbalancer/servers/{subdomain}/url → "http://192.168.1.100:2567"
traefik/http/services/all-servers/loadbalancer/servers/{subdomain}/url → "http://192.168.1.100:2567"
```

#### HTTP プロバイダー

Traefik が Colyseus の `/__traefik` エンドポイントをポーリングして設定を取得する。Keyspace Notifications が使えない環境向け。

```yaml
# traefik-http-provider.yml
providers:
  http:
    endpoint: "http://127.0.0.1:2567/__traefik"
    pollInterval: "5s"
```

### 使用方法

```typescript
import { exposeServerToTraefik } from "@colyseus/traefik";

// サーバー起動後に呼び出す
await exposeServerToTraefik({
  server,
  mainAddress: "backend.yourgamedomain.com",
  provider: "redis",  // または "http"
  internalAddress: "192.168.1.100",  // 省略時は自動検出
});
```

### 内部 IP の自動検出

`internalAddress` を省略すると、`os.networkInterfaces()` から最初の非内部 IPv4 アドレスが自動検出される。

### シャットダウン時のクリーンアップ

`server.onShutdown()` フックで以下を自動実行する：

- Traefik ルーター設定の削除
- サービス設定（個別 + all-servers）の削除
- ヘルスチェック設定の削除

### ルーティングの仕組み

Traefik は 2 種類のルーターを管理する：

1. **個別ルーター**: `Host("node-1.yourgamedomain.com")` → 特定のプロセスへ直接ルーティング（WebSocket 再接続・ルーム参加時）
2. **全体ルーター**: `Host("backend.yourgamedomain.com")` → `all-servers` サービス経由で全プロセスにロードバランシング（初回接続・マッチメイキングリクエスト時）

## 5. PM2 による複数プロセス運用

### NODE_APP_INSTANCE によるポート分離

`@colyseus/tools`（`packages/tools/src/index.ts`）の `listen()` 関数は、`NODE_APP_INSTANCE` 環境変数をポートオフセットとして使用する。

```typescript
// packages/tools/src/index.ts
const processNumber = Number(process.env.NODE_APP_INSTANCE || "0");
port += processNumber;
```

PM2 はインスタンスごとに `NODE_APP_INSTANCE` を 0, 1, 2, ... と自動設定するため、各プロセスが異なるポートで起動する：

| インスタンス | NODE_APP_INSTANCE | ポート |
|-------------|-------------------|--------|
| Process 0 | 0 | 2567 |
| Process 1 | 1 | 2568 |
| Process 2 | 2 | 2569 |
| Process 3 | 3 | 2570 |

### PM2 の設定例

```javascript
// ecosystem.config.js
module.exports = {
  apps: [{
    name: "colyseus",
    script: "dist/index.js",
    instances: 4,            // CPU コア数に応じて調整
    exec_mode: "fork",       // cluster モードは使わない（ポートが重複するため）
    wait_ready: true,        // process.send('ready') を待つ
    env: {
      NODE_ENV: "production",
    }
  }]
};
```

### process.send('ready') の連携

`listen()` 関数は起動完了時に `process.send('ready')` を呼び出す。PM2 の `wait_ready: true` と組み合わせることで、ゼロダウンタイムデプロイが可能になる。

```typescript
// packages/tools/src/index.ts
if (typeof(process.send) === "function") {
    process.send('ready');
}
```

### PM2 メトリクスの自動登録

`Stats.ts` は `@pm2/io` を動的にインポートし、CCU とルーム数をPM2 のメトリクスとして公開する：

```typescript
// packages/core/src/Stats.ts
import('@pm2/io').then((io) => {
    io.default.metric({ id: 'app/stats/ccu', name: 'ccu', value: () => local.ccu });
    io.default.metric({ id: 'app/stats/roomcount', name: 'roomcount', value: () => local.roomCount });
}).catch(() => { });
```

## 6. 推奨アーキテクチャ

### 小規模（~500 CCU）

```
┌─────────────┐
│   Client     │
└──────┬──────┘
       │
┌──────▼──────┐
│  Colyseus   │  ← LocalPresence + LocalDriver
│  (1 process)│  ← ポート 2567
└─────────────┘
```

**構成:**
- 単一プロセス
- LocalPresence + LocalDriver（デフォルト）
- リバースプロキシは任意（Nginx/Caddy 等）

**メリット:** インフラ最小、運用シンプル
**限界:** 1 プロセスの CPU/メモリ上限に依存

### 中規模（500~5,000 CCU）

```
┌─────────────┐
│   Client     │
└──────┬──────┘
       │
┌──────▼──────┐
│   Traefik   │  ← ロードバランサー
└──┬───┬───┬──┘
   │   │   │
┌──▼┐┌─▼┐┌─▼─┐
│P:0││P:1││P:2│  ← PM2 で管理、各プロセスにポート割当
└─┬─┘└─┬─┘└─┬─┘
  └──┬──┘──┬─┘
┌────▼─────▼──┐
│    Redis    │  ← Presence + Driver
└─────────────┘
```

**構成:**
- PM2 で 2~8 プロセス（CPU コア数に応じて）
- RedisPresence + RedisDriver
- Traefik（Redis プロバイダー推奨）
- 単一マシン上で完結

**設定:**

```typescript
import { RedisPresence } from "@colyseus/redis-presence";
import { RedisDriver } from "@colyseus/redis-driver";

const REDIS_URI = "redis://localhost:6379";

export default defineServer({
  rooms: { /* ... */ },
  options: {
    presence: new RedisPresence(REDIS_URI),
    driver: new RedisDriver(REDIS_URI),
    publicAddress: `node-${process.env.NODE_APP_INSTANCE || "0"}.yourgamedomain.com`,
  },
});
```

**注意点:**
- 各プロセスの `publicAddress` は一意にする
- Traefik の Redis Keyspace Notifications を有効にする
- Redis がボトルネックにならないよう監視する

### 大規模（5,000+ CCU）

```
┌──────────────┐
│    Client    │
└──────┬───────┘
       │
┌──────▼───────┐
│   Traefik    │  ← グローバルロードバランサー
└──┬───┬───┬───┘
   │   │   │
┌──▼─┐┌▼──┐┌▼──┐
│VM-A││VM-B││VM-C│  ← 各マシンに PM2 で複数プロセス
│P0  ││P0  ││P0  │
│P1  ││P1  ││P1  │
│P2  ││P2  ││P2  │
└─┬──┘└─┬──┘└─┬──┘
  └──┬──┘──┬──┘
┌────▼─────▼──┐
│ Redis       │  ← Cluster 構成推奨
│ (Cluster)   │
└─────────────┘
```

**構成:**
- 複数マシン、各マシンで PM2 複数プロセス
- RedisPresence + RedisDriver（Redis Cluster）
- Traefik（Redis プロバイダー）
- ルーム生成のプロセス選択を `selectProcessIdToCreateRoom` でカスタマイズ可能

**Redis Cluster の設定:**

```typescript
const clusterNodes = [
  { host: "redis-1.internal", port: 6379 },
  { host: "redis-2.internal", port: 6379 },
  { host: "redis-3.internal", port: 6379 },
];

const presence = new RedisPresence(clusterNodes);
const driver = new RedisDriver(clusterNodes);
```

**カスタムプロセス選択:**

```typescript
// デフォルト: ルーム数が最も少ないプロセスを選択
// カスタム例: 地域・負荷に応じた選択
const server = new Server({
  selectProcessIdToCreateRoom: async (roomName, clientOptions) => {
    const allStats = await matchMaker.stats.fetchAll();
    // カスタムロジック
    return allStats
      .sort((a, b) => a.roomCount - b.roomCount)[0]?.processId;
  },
});
```

### スタンドアロン MatchMaker モード

大規模構成では、マッチメイキング専用プロセスを分離することも可能。`accept(true)` を呼び出すとスタンドアロンモードになり、IPC 購読やヘルスチェックをスキップする。このプロセスはルーム生成を行わず、マッチメイキングの振り分けのみを担当する。

```
MatchMaker.ts (accept):
  if (isStandalone) {
    state = MatchMakerState.READY;
    return;
  }
```

### Colyseus Cloud 構成

`COLYSEUS_CLOUD` 環境変数が設定されている場合、`@colyseus/tools` が自動的に以下を行う：

- CPU が 2 コア以上、または `REDIS_URI` が設定されていれば RedisPresence + RedisDriver を自動構成
- `publicAddress` を `SUBDOMAIN.SERVER_NAME/PORT` 形式で自動生成
- Unix ソケット（`/run/colyseus/{port}.sock`）で listen
- ポートは 2567 に固定

## チェックリスト

### 分散構成移行時の確認事項

- [ ] `npm install @colyseus/redis-presence @colyseus/redis-driver` を実行
- [ ] Redis サーバーが稼働中であること
- [ ] Redis の Keyspace Notifications が有効であること（Redis プロバイダー使用時）
- [ ] 各プロセスに一意な `publicAddress` を設定
- [ ] Traefik の設定ファイルを配置
- [ ] PM2 の `ecosystem.config.js` を作成（`exec_mode: "fork"`、`wait_ready: true`）
- [ ] WebSocket のスティッキーセッションが不要であることを確認（Colyseus は publicAddress ベースで直接ルーティングするため）
