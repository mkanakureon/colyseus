## 1. マッチメイキングアルゴリズムの全体フロー

### 1.1 概要

Colyseus のマッチメイキングは `MatchMaker` モジュール（`packages/core/src/MatchMaker.ts`）が中心となり、以下の流れで動作する。

```
クライアント → HTTP POST /matchmake/{method}/{roomName}
            → controller.invokeMethod()
            → matchMaker.joinOrCreate() / create() / join() / joinById() / reconnect()
            → Driver でルームキャッシュを検索・作成
            → Seat Reservation（席予約）を返却
            → クライアントが WebSocket で接続
```

### 1.2 初期化フロー

1. `matchMaker.setup()` で Presence、Driver、publicAddress を初期化
2. `matchMaker.accept()` で IPC チャネルを購読開始（リモートルーム作成リクエストの受付）
3. 起動時にヘルスチェック（`healthCheckAllProcesses()`）で残存する無効なプロセスを除去
4. `stats.setAutoPersistInterval()` で 1 分ごとにプロセス統計を永続化

### 1.3 ルーム定義

```typescript
gameServer.define("room_name", RoomClass, defaultOptions)
```

内部では `defineRoomType()` が `RegisteredHandler` を生成し、`handlers` マップに名前をキーとして登録する。

### 1.4 Seat Reservation（席予約）パターン

全てのマッチメイキングメソッドは最終的に `ISeatReservation` を返す。

```typescript
interface ISeatReservation {
  name: string;        // ルーム名
  sessionId: string;   // クライアントの一意セッション ID
  roomId: string;      // ルームインスタンス ID
  processId: string;   // ルームが動作するプロセス ID
  publicAddress?: string;
  devMode?: boolean;
}
```

席予約後、クライアントは `sessionId` と `roomId` を使って WebSocket 接続を確立する。

## 2. Room 検索・フィルタリングの仕組み

### 2.1 filterBy

`RegisteredHandler.filterBy()` で、ルーム検索時に使用するフィルタフィールドを定義する。

```typescript
gameServer.define("battle", BattleRoom)
  .filterBy(["mode", "difficulty", "maxClients"]);
```

**動作原理:**

1. クライアントが `{ mode: "ranked", difficulty: "hard" }` を送信
2. `handler.getFilterOptions(clientOptions)` がフィルタ対象フィールドのみ抽出
3. `driver.findOne()` に `{ locked: false, name: "battle", private: false, mode: "ranked", difficulty: "hard" }` として渡される
4. Driver がこの条件に一致するルームを返す

**無効なフィールド:** `clients`, `locked`, `private`, `metadata`, `name`, `processId`, `roomId` はフィルタオプションとして使用すると警告が出力され無視される（`INVALID_OPTION_KEYS`）。

### 2.2 sortBy

`RegisteredHandler.sortBy()` でルーム検索結果のソート順を定義する。

```typescript
gameServer.define("battle", BattleRoom)
  .sortBy({ clients: -1 });  // クライアント数降順（満室に近いルームを優先）
```

ソートは IRoomCache のフィールドおよび metadata のフィールドの両方に対応する。方向指定は `1 | -1 | 'asc' | 'desc' | 'ascending' | 'descending'`。

### 2.3 metadata とフィルタの関係

`filterBy` で指定されたフィールドは、ルーム作成時に `getMetadataFromOptions()` で自動的に metadata に格納される。`sortBy` で指定されたフィールドも同様に metadata に含まれる。これにより、Driver 側でクエリ時に metadata フィールドでのフィルタリング・ソートが可能になる。

### 2.4 Query クラス（LocalDriver）

`LocalDriver` および `RedisDriver` は `Query` クラスを使ってインメモリでフィルタリングとソートを行う。

- **フィルタ:** 全条件を AND で適用。IRoomCache 直接フィールドと metadata フィールドの両方を検索
- **ソート:** フィルタ後にソートを適用（パフォーマンス最適化）
- **findOne:** フィルタ・ソート後の最初の 1 件を返す（`then()` メソッドで Promise 互換）

### 2.5 findOneRoomAvailable

```typescript
async function findOneRoomAvailable(roomName, filterOptions, additionalSortOptions?)
```

このメソッドは `join` と `joinOrCreate` の内部で使用される。以下の固定条件が自動付与される:

- `locked: false` — ロックされていないルーム
- `private: false` — プライベートでないルーム
- `name: roomName` — 指定されたルーム名

## 3. joinOrCreate / create / join / joinById の違い

### 3.1 一覧比較

| メソッド | ルーム検索 | ルーム作成 | リトライ | 用途 |
|----------|-----------|-----------|---------|------|
| `joinOrCreate` | あり | 見つからなければ作成 | 最大5回（SeatReservationError時） | 一般的なマッチメイキング |
| `create` | なし | 常に作成 | なし | プライベートルーム・カスタムルーム |
| `join` | あり | なし | デフォルトリトライ | 既存ルームへの参加のみ |
| `joinById` | roomId 指定 | なし | なし | 特定ルームへの参加 |
| `reconnect` | roomId 指定 | なし | なし | 再接続（reconnectionToken 必須） |

### 3.2 joinOrCreate の詳細フロー

```
1. onAuth 呼び出し（静的 onAuth が定義されている場合）
2. findOneRoomAvailable() で既存ルーム検索
3. ルームが見つかった場合 → reserveSeatFor() で席予約
4. ルームが見つからない場合:
   a. filterOptions からコンカレンシーキーを生成
   b. concurrentJoinOrCreateRoomLock() で排他制御
      - 同じフィルタ条件の同時リクエストは Presence pub/sub で待機
      - 最初のリクエストのみがルームを作成
      - 作成完了後、roomId を他の待機リクエストに通知
   c. createRoom() でルーム作成
5. reserveSeatFor() で席予約
6. SeatReservationError の場合、最大5回リトライ
```

**同時作成防止メカニズム:**

`concurrentJoinOrCreateRoomLock()` は Presence の `hincrbyex` を使い、同一フィルタ条件での同時ルーム作成を防ぐ。最初のリクエストがルームを作成し、後続のリクエストは `concurrent:{roomName}:{concurrencyKey}` チャネルで roomId の通知を待機する。

### 3.3 create

```
1. onAuth 呼び出し
2. createRoom() で必ず新規ルームを作成
3. reserveSeatFor() で席予約
```

### 3.4 join

```
1. onAuth 呼び出し
2. findOneRoomAvailable() で検索
3. 見つからなければ ServerError (MATCHMAKE_INVALID_CRITERIA) をスロー
4. reserveSeatFor() で席予約
5. SeatReservationError の場合リトライ
```

### 3.5 joinById

```
1. driver.findOne({ roomId }) で直接検索
2. 見つからなければ ServerError (MATCHMAKE_INVALID_ROOM_ID)
3. ロック中なら ServerError (MATCHMAKE_INVALID_ROOM_ID)
4. onAuth 呼び出し
5. reserveSeatFor() で席予約
```

### 3.6 reconnect

```
1. driver.findOne({ roomId }) で検索
2. reconnectionToken の検証（remoteRoomCall で Room.checkReconnectionToken 呼び出し）
3. 有効なら buildSeatReservation() を返却
4. 無効なら ServerError (MATCHMAKE_EXPIRED)
```

### 3.7 席予約プロセス（reserveSeatFor）

```typescript
async function reserveSeatFor(room, options, authData?)
```

1. sessionId を生成（authData に sessionId があればそれを使用）
2. `remoteRoomCall(roomId, '_reserveSeat', [sessionId, options, authData])` でルームに席予約リクエスト
3. ローカルルームならメソッド直接呼び出し、リモートなら IPC 経由
4. 失敗時: IPC タイムアウトならヘルスチェック実行、プロセスが死んでいれば `SeatReservationError`
5. 成功時: `ISeatReservation` を構築して返却

## 4. Driver インターフェースとストレージ戦略

### 4.1 MatchMakerDriver インターフェース

`packages/core/src/matchmaker/driver.ts` に定義。全ドライバーが実装するインターフェース:

| メソッド | 説明 |
|----------|------|
| `has(roomId)` | ルームの存在確認 |
| `query(conditions, sortOptions?)` | 条件に合うルームリストを取得 |
| `findOne(conditions, sortOptions?)` | 条件に合う最初のルームを取得 |
| `update(room, operations)` | `$set` / `$inc` オペレーションでルーム更新 |
| `persist(room, create?)` | ルームをキャッシュに永続化（create=true で新規作成） |
| `remove(roomId)` | ルームをキャッシュから削除 |
| `cleanup(processId)` | プロセス ID に紐づくルームを一括削除 |
| `boot()` | ドライバー初期化（オプション） |
| `shutdown()` | ドライバー接続終了 |
| `clear()` | 全データ削除（テスト用） |

### 4.2 IRoomCache 構造

```typescript
interface IRoomCache<Metadata = any> {
  name: string;           // ルーム名
  roomId: string;         // 一意識別子
  processId: string;      // 処理プロセス ID
  clients: number;        // 現在の接続数
  maxClients: number;     // 最大接続数
  locked?: boolean;       // ロック状態
  private?: boolean;      // プライベートフラグ
  publicAddress?: string; // 公開アドレス
  unlisted?: boolean;     // ロビー非表示フラグ
  metadata?: Metadata;    // カスタムメタデータ
  createdAt?: Date;       // 作成日時
}
```

### 4.3 LocalDriver（デフォルト）

- **ストレージ:** インメモリ配列（`rooms: IRoomCache[]`）
- **検索:** `Query` クラスによるインメモリフィルタリング・ソート
- **用途:** 単一プロセスでの開発・テスト
- **永続化:** `persist()` は `create=true` のときのみ配列に追加。`create=false` では何もしない（インメモリなのでオブジェクト参照がそのまま更新される）
- **制限:** プロセス間でルーム情報を共有できない

### 4.4 RedisDriver

- **ストレージ:** Redis Hash（キー: `roomcaches`、フィールド: `roomId`、値: JSON 文字列）
- **検索:** 全ルームを `HGETALL` で取得し、`Query` クラスでインメモリフィルタリング
- **最適化:**
  - 同時リクエスト時の `HGETALL` 重複防止（`_concurrentRoomCacheRequest`）
  - ルーム名での事前フィルタ（JSON パース前に文字列マッチ）
  - バッチ削除（500 件ずつ `HDEL`）
- **用途:** 分散マッチメイキング（複数プロセス間でルーム情報を共有）
- **接続:** Redis 単体または Redis Cluster に対応

### 4.5 MongooseDriver

- **ストレージ:** MongoDB コレクション（`RoomCache` モデル）
- **スキーマ:** `strict: false` で柔軟なメタデータ対応
- **インデックス:** `{ name: 1, locked: -1 }` と `{ roomId: 1 }`
- **検索:** MongoDB ネイティブクエリ（`find()` / `findOne()`）とソート
- **用途:** MongoDB を既に使用しているプロジェクト

### 4.6 PostgresDriver（Drizzle）

- **ストレージ:** PostgreSQL テーブル（`roomcaches_v1`）
- **ORM:** Drizzle ORM（`drizzle-orm/postgres-js`）
- **スキーマ:** `pgTable` で定義、カスタムスキーマの注入も可能
- **検索:** SQL クエリ（WHERE 句と ORDER BY の動的構築）
- **boot():** `CREATE TABLE IF NOT EXISTS` でテーブル自動作成
- **アトミック更新:** クライアント数増加時に `locked` フラグを SQL 内で原子的に計算（レースコンディション防止）
  ```sql
  locked = locked OR ((clients + increment) >= maxClients)
  ```
- **用途:** PostgreSQL を使用する本番環境

### 4.7 ドライバー選択戦略

| 環境 | 推奨ドライバー | 理由 |
|------|--------------|------|
| ローカル開発（単一プロセス） | LocalDriver | 外部依存なし |
| 本番（単一プロセス） | LocalDriver | 最低レイテンシー |
| 本番（複数プロセス） | RedisDriver | Presence と共有可能 |
| 本番（PostgreSQL 環境） | PostgresDriver | DB 一元化 |
| レガシー | MongooseDriver | MongoDB 既存環境 |

## 5. 分散マッチメイキング（IPC, Redis Driver）

### 5.1 IPC（Inter-Process Communication）

Colyseus は Presence の pub/sub を利用したプロセス間通信（IPC）で分散マッチメイキングを実現する。

**チャネル規約:**

| チャネル | 形式 | 用途 |
|---------|------|------|
| プロセスチャネル | `p:{processId}` | ルーム作成リクエスト・ヘルスチェック |
| ルームチャネル | `${roomId}` | ルームメソッド呼び出し（_reserveSeat 等） |
| IPC レスポンス | `ipc:{requestId}` | リクエストごとの一意レスポンスチャネル |
| 同時作成通知 | `concurrent:{roomName}:{key}` | joinOrCreate 同時リクエスト調整 |
| ロビー通知 | `$lobby` | ルーム状態変更のリアルタイム通知 |

### 5.2 IPC プロトコル

```
リクエスト:  presence.publish(channel, [method, requestId, args])
レスポンス:  presence.publish("ipc:{requestId}", [IpcProtocol.SUCCESS, data])
            presence.publish("ipc:{requestId}", [IpcProtocol.ERROR, errorMessage])
タイムアウト: REMOTE_ROOM_SHORT_TIMEOUT ミリ秒で "ipc_timeout" エラー
```

### 5.3 リモートルーム作成フロー

```
1. selectProcessIdToCreateRoom() でプロセス選択
   - デフォルト: roomCount が最小のプロセス
   - カスタムコールバックで上書き可能
2. 選択されたプロセスが自プロセスの場合:
   → handleCreateRoom() を直接呼び出し
3. 選択されたプロセスが別プロセスの場合:
   → requestFromIPC() で "p:{selectedProcessId}" に送信
   → リモートプロセスの subscribeIPC ハンドラが handleCreateRoom() を実行
   → 結果を IPC レスポンスで返却
4. タイムアウト時:
   → 対象プロセスを stats から除外
   → 自プロセスでフォールバック作成
```

### 5.4 リモートルームメソッド呼び出し

`remoteRoomCall()` は、ルームがローカルにあればメソッド直接呼び出し、なければ IPC 経由で呼び出す。

```typescript
// ローカル
room[method].apply(room, args);

// リモート
requestFromIPC(presence, "$roomId", method, args, timeout);
```

主な利用箇所:
- `_reserveSeat` — 席予約
- `_reserveMultipleSeats` — 複数席予約（QueueRoom 用）
- `checkReconnectionToken` — 再接続トークン検証

### 5.5 ヘルスチェック

**起動時:** `healthCheckAllProcesses()` が全プロセスの死活を確認。レスポンスのないプロセスは stats から除外し、そのプロセスのルームキャッシュを `driver.cleanup()` で削除。

**ランタイム:** 席予約の IPC タイムアウト時に `healthCheckProcessId()` を呼び出し、不応答プロセスを除外。同一プロセスへの重複ヘルスチェックは Promise 共有で防止。

### 5.6 プロセス統計（Stats）

各プロセスは `roomCount` と `ccu`（同時接続数）を Presence Hash（キー: `roomcount`）に永続化する。

- **永続化頻度:** 最大 1 秒に 1 回（スロットリング）
- **自動永続化:** 1 分間隔（ヘルスチェックの一部として）
- **ロードバランシング:** `selectProcessIdToCreateRoom` が `roomCount` 昇順で最小のプロセスを選択

## 6. QueueRoom によるキューイング

### 6.1 概要

`QueueRoom`（`packages/core/src/rooms/QueueRoom.ts`）は、ランク制マッチメイキング（SBMM: Skill-Based Matchmaking）を実現するための専用ルーム。通常の `joinOrCreate` とは異なり、プレイヤーをキューに溜めてから適切なグループを形成し、別のゲームルームにシートを予約する。

### 6.2 設定オプション（QueueOptions）

| オプション | 型 | デフォルト | 説明 |
|-----------|------|----------|------|
| `matchRoomName` | `string` | **必須** | マッチ成立時に作成するルーム名 |
| `maxPlayers` | `number` | 4 | 1 マッチのプレイヤー数 |
| `maxWaitingCycles` | `number` | 15 | この周期を超えると不完全グループでもマッチ成立（`allowIncompleteGroups` 時） |
| `maxWaitingCyclesForPriority` | `number` | 10 | この周期を超えるとランク不一致でも強制マッチ |
| `maxTeamSize` | `number` | undefined | チーム制マッチ時のチームサイズ |
| `allowIncompleteGroups` | `boolean` | false | maxPlayers 未満でもマッチ成立を許可（Bot 補充想定） |
| `compare` | `function` | ランク差10以下またはランク比2倍以下 | プレイヤーとグループの互換性判定 |
| `onGroupReady` | `function` | `matchMaker.createRoom(matchRoomName)` | グループ完成時のカスタムルーム作成 |

### 6.3 マッチメイキングサイクル

`cycleTickInterval`（デフォルト 1000ms）ごとに `reassignMatchGroups()` が実行される。

```
1. 全グループをリセット
2. クライアントをランク昇順でソート（"ready" 状態のグループに属するクライアントは除外）
3. チーム制かどうかで分岐:
   - チーム制: redistributeTeams()
   - 個人制: redistributeClients()
4. 高優先度グループの評価（evaluateHighPriorityGroups）
5. 準備完了グループの処理（processGroupsReady）
```

### 6.4 グループ形成アルゴリズム（redistributeClients）

```
ソート済みクライアントを順に処理:
  1. currentCycle をインクリメント
  2. compare() でグループとの互換性を判定
     - 互換性なし & highPriority でない → 新グループ作成
     - 互換性あり or highPriority → 現グループに追加
  3. 平均ランクを更新
  4. グループが maxPlayers に達したら → ready = true
  5. maxWaitingCycles 超過 & allowIncompleteGroups → highPriorityGroups に追加
  6. maxWaitingCyclesForPriority 超過 → highPriority = true（次サイクルでランク無視マッチ）
```

**デフォルト compare 関数:**
```typescript
(client, matchGroup) => {
  const diff = Math.abs(client.rank - matchGroup.averageRank);
  const diffRatio = diff / matchGroup.averageRank;
  return (diff < 10 || diffRatio <= 2);
}
```

### 6.5 チーム制マッチメイキング（redistributeTeams）

1. クライアントを `teamId` でグループ化
2. 各チームの平均ランクを計算
3. チームをランク昇順でソート
4. `maxTeamSize` 単位でチームからメンバーを切り出し、`redistributeClients` でグループに配置
5. チームの残りメンバーが `maxTeamSize` 未満になったら次サイクルへ

### 6.6 グループ完成時の処理（processGroupsReady）

```
ready なグループごとに:
  1. onGroupReady() でゲームルームを作成
  2. reserveMultipleSeatsFor() で全メンバーの席を一括予約
  3. 各クライアントに "seat" メッセージで ISeatReservation を送信
  4. 失敗時: 全クライアントを leave（クライアント側で再キューは任意）

ready でないグループ:
  - "clients" メッセージでキュー内の人数を通知（変更時のみ）
```

### 6.7 クライアントメッセージ

| メッセージ | 方向 | 内容 |
|-----------|------|------|
| `"clients"` | サーバー → クライアント | キュー内の人数 |
| `"seat"` | サーバー → クライアント | ISeatReservation（マッチ成立時） |
| `"confirm"` | クライアント → サーバー | 接続確認（クライアントが leave） |

## 7. ルーム数制限・自動スケーリング

### 7.1 プロセスレベルのルーム数管理

- `stats.local.roomCount` でプロセスごとのルーム数を追跡
- `handleCreateRoom()` でルーム作成時にインクリメント
- `disposeRoom()` でルーム破棄時にデクリメント
- `roomCount` が 0 になると `'no-active-rooms'` イベントを発火（グレースフルシャットダウンで使用）

### 7.2 ロードバランシング（selectProcessIdToCreateRoom）

デフォルト実装:

```typescript
selectProcessIdToCreateRoom = async function () {
  return (await stats.fetchAll())
    .sort((p1, p2) => p1.roomCount > p2.roomCount ? 1 : -1)[0]
    ?.processId || processId;
};
```

- 全プロセスの `roomCount` を取得し、最小のプロセスを選択
- カスタムコールバックで上書き可能（`setup()` の第4引数）

### 7.3 ルームのロック・アンロック

- ルームの `maxClients` に達するとルームが自動ロック（`_reserveSeat` 内）
- ロックされたルームは `findOneRoomAvailable()` の `locked: false` 条件で検索から除外
- `room.lock()` で手動ロック、`room.unlock()` で手動アンロック可能
- PostgresDriver ではロック判定を SQL 内でアトミックに計算（レースコンディション防止）

### 7.4 ルームの可視性制御

| プロパティ | 効果 |
|-----------|------|
| `private: true` | `join()` / `joinOrCreate()` の検索から除外。`joinById()` でのみ参加可能 |
| `locked: true` | 全てのマッチメイキング検索から除外 |
| `unlisted: true` | ロビーリスティングから除外（マッチメイキング検索には影響しない） |

### 7.5 グレースフルシャットダウン

```
1. stats から自プロセスを除外（新規ルーム作成停止）
2. 全ローカルルームをロック（新規参加停止）
3. 各ルームの onBeforeShutdown() を呼び出し
4. 全ルームが dispose されるまで待機
5. driver.cleanup() でルームキャッシュを削除
6. IPC チャネルの購読解除
7. 全クライアントを切断
```

### 7.6 Lobby（リアルタイムルームリスト）

`RegisteredHandler.enableRealtimeListing()` を有効にすると、ルームの作成・ロック・アンロック・参加・離脱・可視性変更・メタデータ変更・破棄の各イベントで Lobby チャネル（`$lobby`）に通知が発行される。

`subscribeLobby()` でこの通知を購読し、クライアントにリアルタイムなルーム一覧を提供できる。

### 7.7 自動スケーリングの考慮事項

Colyseus 自体にはオートスケーリング機能は組み込まれていないが、以下の仕組みで外部オートスケーラーとの統合が可能:

- **プロセス統計:** `stats.fetchAll()` で全プロセスの `roomCount` と `ccu` を取得可能
- **グローバル CCU:** `stats.getGlobalCCU()` で全プロセスの合計 CCU を取得
- **スタンドアロンモード:** `accept(true)` でマッチメイキング専用プロセスとして起動可能（ルーム生成しない）
- **カスタムプロセス選択:** `selectProcessIdToCreateRoom` をカスタマイズして、外部メトリクスに基づくプロセス選択が可能
- **ヘルスチェック:** 不応答プロセスの自動除外で、スケールイン時の整合性を維持
