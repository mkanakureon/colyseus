## 1. 状態同期の全体アーキテクチャ

Colyseus の状態同期は、**サーバー権威型（Server-Authoritative）** のアーキテクチャを採用している。サーバーが唯一の状態の真実源（Single Source of Truth）であり、クライアントはサーバーから送られるパッチを適用して状態を再構築する。

### 全体フロー

```
[サーバー Room]
    |
    |-- state = new MyState()  (Schema インスタンスをセット)
    |
    |-- patchRate (デフォルト 50ms / 20fps) の間隔で broadcastPatch() を実行
    |       |
    |       |-- onBeforePatch() コールバック（任意）
    |       |-- _serializer.applyPatches(clients, state)
    |       |       |-- 変更検出 (encoder.hasChanges)
    |       |       |-- デルタエンコード (encoder.encode)
    |       |       |-- フィルタ処理（StateView がある場合）
    |       |       |-- client.raw(encodedChanges) で各クライアントに送信
    |       |-- _dequeueAfterPatchMessages()
    |
    v
[WebSocket / HTTP3]
    |
    v
[クライアント SDK Room]
    |-- onMessageCallback で Protocol.ROOM_STATE_PATCH を受信
    |-- serializer.patch(buffer, it) でデコード
    |-- onStateChange シグナルを発火
```

### 接続時の初期同期

1. クライアントが Room に接続すると、サーバーは `Protocol.JOIN_ROOM` メッセージを送信（reconnectionToken + serializerId + handshake バイト列を含む）
2. クライアントが `Protocol.JOIN_ROOM` で ACK を返す
3. サーバー側で `client.state` が `ClientState.JOINED` に遷移
4. サーバーが `sendFullState(client)` で `Protocol.ROOM_STATE` として完全な状態を送信
5. 以降は `broadcastPatch()` によるデルタ同期に移行

### シミュレーションとパッチの分離

Colyseus はゲームループ（シミュレーション）と状態送信（パッチ）を独立したインターバルで管理する。

| インターバル | デフォルト値 | 役割 |
|-------------|------------|------|
| `setSimulationInterval` | 16.66ms (60fps) | ゲームロジックの更新（`clock.tick()` + コールバック） |
| `patchRate` | 50ms (20fps) | 状態変更の検出とクライアントへのブロードキャスト |

これにより、ゲームロジックは 60fps で動作しつつ、ネットワーク送信は 20fps に抑えるという構成が可能になる。`patchRate` を `null` または `0` に設定すると自動パッチ送信を無効化でき、手動で `broadcastPatch()` を呼ぶ運用も可能。

## 2. @colyseus/schema との連携

### 外部パッケージとしての位置づけ

`@colyseus/schema` は Colyseus 本体とは別の独立パッケージであり、以下の機能を提供する。

- **スキーマ定義**: デコレータベース（`@type()` 等）の型定義
- **Encoder / Decoder**: バイナリシリアライゼーション・デシリアライゼーション
- **Reflection**: スキーマ構造をバイト列にエンコードし、クライアントへハンドシェイクとして送信
- **StateView**: フィルタリング機能（クライアントごとに異なる状態ビューを提供）
- **$changes**: 変更追跡メカニズム

### サーバー側の連携

```typescript
// packages/core/src/Room.ts より
// state セッター内で Schema を自動検出
set: (newState) => {
    if (newState?.constructor[Symbol.metadata] !== undefined || newState[$changes] !== undefined) {
        this.setSerializer(new SchemaSerializer());
    }
    this._serializer.reset(newState);
}
```

`state` プロパティに Schema インスタンスを代入すると、`$changes` または `Symbol.metadata` の存在を検出して自動的に `SchemaSerializer` が選択される。

### サーバー側 SchemaSerializer の主要操作

| メソッド | 処理内容 |
|---------|---------|
| `reset(newState)` | `Encoder` を初期化。フィルタの有無を検出 |
| `getFullState(client?)` | 完全な状態をエンコード。フィルタがある場合は `encodeAllView` を使用 |
| `applyPatches(clients)` | 変更をエンコードし、全クライアントに送信。フィルタ有無で処理が分岐 |
| `handshake()` | `Reflection.encode()` でスキーマ構造をキャッシュして返す |

### クライアント側の連携

クライアント SDK の `SchemaSerializer` は `Decoder` を使って受信バイト列をデコードする。

```typescript
// packages/sdk/src/serializer/SchemaSerializer.ts
setState(encodedState, it?) { this.decoder.decode(encodedState, it); }
patch(patches, it?)        { return this.decoder.decode(patches, it); }
handshake(bytes, it?) {
    // ローカルに state がある場合: Reflection を no-op で消費
    // ない場合: Reflection.decode でサーバーのスキーマから state を生成
}
```

## 3. パッチサイクル

### broadcastPatch の実行フロー

`broadcastPatch()` は `patchRate` の間隔で自動呼び出しされる（`setInterval`）。

```
broadcastPatch()
  |
  |-- onBeforePatch(state)        // ユーザー定義の前処理
  |-- clock.tick()                // シミュレーションインターバルがない場合のみ
  |-- _serializer.applyPatches(clients, state)
  |       |
  |       |-- encoder.hasChanges をチェック
  |       |-- 変更なし → discardChanges() して return false
  |       |-- 変更あり:
  |       |     |-- encoder.encode(it) でデルタバイト列を生成
  |       |     |-- Protocol.ROOM_STATE_PATCH (15) をヘッダーに付与
  |       |     |-- フィルタなし: 全クライアントに同一バイト列を送信
  |       |     |-- フィルタあり: クライアントごとに encodeView で個別エンコード
  |       |     |-- encoder.discardChanges() で変更フラグをリセット
  |       |
  |       +-- return hasChanges
  |
  |-- _dequeueAfterPatchMessages()  // afterNextPatch キューの処理
  +-- return hasChanges
```

### patchRate の設定

```typescript
// デフォルト: 50ms (20fps)
const DEFAULT_PATCH_RATE = 1000 / 20;

// patchRate セッターの動作
set patchRate(milliseconds) {
    // 既存の interval をクリア
    // milliseconds が null/0 でなければ setInterval で broadcastPatch を設定
    // milliseconds が null/0 かつ simulationInterval もない場合は clock.tick のみ
}
```

### デルタ圧縮の仕組み

Schema シリアライザでは `@colyseus/schema` の `Encoder` が変更追跡を行う。

1. Schema オブジェクトのプロパティ変更時に `$changes` が自動更新される
2. `encoder.encode()` は変更されたフィールドのみをバイナリエンコードする
3. エンコード後 `discardChanges()` で変更フラグをリセット
4. クライアント側は `Decoder.decode()` で受信バイト列を既存の状態オブジェクトに適用

変更がない場合（`encoder.hasChanges === false`）、パッチは送信されない。ただし StateView のフィルタ変更がある場合は、状態自体に変更がなくても該当クライアントにパッチが送信される。

### fullEncodeBuffer の最適化

サーバー側 `SchemaSerializer` は `fullEncodeBuffer` を保持し、`needFullEncode` フラグで再エンコードの必要性を管理する。新規クライアントへの fullState 送信時、変更がなければキャッシュを再利用する。

## 4. シリアライゼーション戦略

Colyseus は複数のシリアライゼーション戦略をサポートしている。

### 4.1 Schema Serializer（推奨・デフォルト）

**ID:** `schema`（サーバー側 `packages/core/src/serializer/SchemaSerializer.ts`）

- `@colyseus/schema` のバイナリエンコーダを使用
- フィールド単位の変更追跡による効率的なデルタ圧縮
- `Reflection` によるハンドシェイク（スキーマ構造の自動共有）
- `StateView` によるクライアント別フィルタリング
- 型安全なコールバック（`getStateCallbacks` / `getDecoderStateCallbacks`）

**サーバー側の特徴:**
- `Encoder` クラスで状態をエンコード
- `sharedBuffer` を使い、同一パッチは一度だけエンコード
- フィルタありの場合は `encodeView` でクライアントごとにエンコード（`encodedViews` の Map でキャッシュ）

**クライアント側の特徴:**
- `Decoder` クラスで受信バイト列をデコード
- ハンドシェイクが `Reflection.decode` でスキーマ構造を復元（ローカルスキーマがない場合）
- ローカルスキーマがある場合はハンドシェイクを no-op で消費し、ローカル定義を優先

### 4.2 Legacy Schema Serializer

**ID:** `schema-v2`（`packages/serializer/legacy-schema-serializer/src/index.ts`）

- `@colyseus/schema` の旧 API（`state.encodeAll()`, `state.encode()`, `state.$changes`）を使用
- `hasFilter()` でフィルタの有無を判定
- フィルタありの場合 `state.applyFilters(client)` でクライアント別エンコード
- 現行 SchemaSerializer との違いは、`Encoder` クラスを介さず `Schema` インスタンスの直接メソッドを使う点

### 4.3 FossilDelta Serializer

**ID:** `fossil-delta`（`packages/serializer/fossil-delta-serializer/src/index.ts`）

- `fossil-delta` ライブラリによるバイナリ差分圧縮
- `msgpackr` で状態全体を msgpack エンコードし、前回との差分を `fossilDelta.create()` で算出
- Schema 以外の任意の JavaScript オブジェクトを状態として使用可能
- `$changes` がある場合（Schema）は変更サイズで最適化判定

**クライアント側（コメントアウト済み）:**
- `@gamestdio/state-listener` の `StateContainer` を使用
- `fossilDelta.apply()` でパッチ適用後、`msgpack.decode` で復元
- 現在は SDK のクライアント側実装はコメントアウトされており、レガシー扱い

### 4.4 None Serializer

**ID:** `none`

- 状態同期を行わないダミー実装
- `state` を設定しない Room（メッセージパッシングのみ）で使用される
- `applyPatches()` は常に `false` を返す

### 戦略の自動選択

サーバー側では `state` セッターで Schema インスタンスの検出が行われ、自動的に `SchemaSerializer` が選択される。Schema を使わない場合は `NoneSerializer` がデフォルト。FossilDelta は別途明示的にセットする必要がある。

クライアント側では、サーバーから `JOIN_ROOM` メッセージで送られる `serializerId` 文字列に基づいて `getSerializer()` でシリアライザが選択される。

## 5. メッセージプロトコル

### Protocol enum

`@colyseus/shared-types` パッケージで定義される共有プロトコルコード。

| コード | 名前 | 値 | 方向 | 説明 |
|--------|------|-----|------|------|
| `JOIN_ROOM` | 10 | 10 | S->C / C->S | ルーム参加通知 / ACK |
| `ERROR` | 11 | 11 | S->C | エラー通知（code + message） |
| `LEAVE_ROOM` | 12 | 12 | S->C / C->S | 退出通知 / 退出要求 |
| `ROOM_DATA` | 13 | 13 | 双方向 | カスタムメッセージ（msgpack） |
| `ROOM_STATE` | 14 | 14 | S->C | 完全な状態（初期同期・再接続時） |
| `ROOM_STATE_PATCH` | 15 | 15 | S->C | デルタパッチ |
| `ROOM_DATA_SCHEMA` | 16 | 16 | - | 非推奨（旧 Schema インスタンス送信用） |
| `ROOM_DATA_BYTES` | 17 | 17 | 双方向 | カスタムメッセージ（生バイト列） |
| `PING` | 18 | 18 | 双方向 | RTT 計測用 |

プロトコルコードは 0〜100 の範囲に収まるよう設計されており、1 バイトでエンコード可能（0〜127）。

### メッセージバイナリフォーマット

全メッセージの先頭 1 バイトがプロトコルコード。以降のペイロードはメッセージ種別により異なる。

**JOIN_ROOM (S->C):**
```
[10] [reconnectionToken長] [reconnectionToken(UTF-8)] [serializerId長] [serializerId(UTF-8)] [handshakeバイト列...]
```

**ERROR (S->C):**
```
[11] [code(数値エンコード)] [message(文字列エンコード)]
```

**ROOM_STATE (S->C):**
```
[14] [完全な状態バイト列...]
```

**ROOM_STATE_PATCH (S->C):**
```
[15] [デルタパッチバイト列...]
```

**ROOM_DATA (双方向):**
```
[13] [メッセージ型(文字列 or 数値エンコード)] [msgpackペイロード...]
```

**ROOM_DATA_BYTES (双方向):**
```
[17] [メッセージ型(文字列 or 数値エンコード)] [生バイト列...]
```

### エンコードの詳細

数値とフォーマットの判定には `@colyseus/schema` の `encode` / `decode` ユーティリティを使用。msgpack ペイロードには `@colyseus/msgpackr`（`Packr` クラス）を使用し、共有バッファ上でゼロコピーに近いエンコードを行う。

### CloseCode

WebSocket クローズコードも `@colyseus/shared-types` で定義される。

| コード | 名前 | 説明 |
|--------|------|------|
| 1000 | `NORMAL_CLOSURE` | 正常切断 |
| 1001 | `GOING_AWAY` | サーバーシャットダウン等 |
| 1005 | `NO_STATUS_RECEIVED` | ステータスなし切断 |
| 1006 | `ABNORMAL_CLOSURE` | 異常切断 |
| 4000 | `CONSENTED` | クライアントが同意して退出 |
| 4001 | `SERVER_SHUTDOWN` | サーバーシャットダウン |
| 4002 | `WITH_ERROR` | エラーによる切断 |
| 4003 | `FAILED_TO_RECONNECT` | 再接続失敗 |
| 4010 | `MAY_TRY_RECONNECT` | 再接続可能 |

クライアント SDK は `GOING_AWAY`, `NO_STATUS_RECEIVED`, `ABNORMAL_CLOSURE`, `MAY_TRY_RECONNECT` の 4 コードで自動再接続を試行する。

## 6. クライアント側の状態デシリアライゼーション

### メッセージ受信フロー

クライアント SDK の `Room.onMessageCallback` が全メッセージのエントリーポイント。

```typescript
// packages/sdk/src/Room.ts - onMessageCallback
const code = buffer[0];

if (code === Protocol.ROOM_STATE) {
    this.serializer.setState(buffer, it);    // 完全な状態を設定
    this.onStateChange.invoke(this.serializer.getState());

} else if (code === Protocol.ROOM_STATE_PATCH) {
    this.serializer.patch(buffer, it);       // デルタパッチを適用
    this.onStateChange.invoke(this.serializer.getState());
}
```

### SchemaSerializer のデコード処理

```typescript
// packages/sdk/src/serializer/SchemaSerializer.ts
setState(encodedState, it?) {
    this.decoder.decode(encodedState, it);
}

patch(patches, it?) {
    return this.decoder.decode(patches, it);
}
```

`setState` と `patch` はどちらも `Decoder.decode()` を呼ぶ。`@colyseus/schema` の Decoder は、受信バイト列の内容に応じて完全な状態復元かインクリメンタルな適用かを自動判別する。

### ハンドシェイクフロー

1. サーバーが `Reflection.encode(encoder)` でスキーマ構造をバイト列化
2. `JOIN_ROOM` メッセージのペイロードとしてクライアントに送信
3. クライアント側:
   - ローカルに `rootSchema` が提供されている場合: `Reflection.decode` を空消費し、ローカルスキーマで `Decoder` を初期化
   - ローカルスキーマがない場合: `Reflection.decode` がスキーマ構造から動的に `state` と `Decoder` を生成

ローカルスキーマを使う場合（`skipHandshake` フラグ）、ハンドシェイクバイト列自体をスキップできる最適化も実装されている。

### シリアライザの登録と選択

```typescript
// packages/sdk/src/serializer/Serializer.ts
const serializers: { [id: string]: any } = {};

export function registerSerializer(id: string, serializer: any) {
    serializers[id] = serializer;
}

export function getSerializer(id: string) {
    const serializer = serializers[id];
    if (!serializer) { throw new Error("missing serializer: " + id); }
    return serializer;
}
```

クライアントは `JOIN_ROOM` メッセージ内の `serializerId` 文字列（例: `"schema"`, `"fossil-delta"`, `"none"`）で適切なシリアライザを取得する。

### 再接続時の状態復元

再接続時（`joinedAtTime > 0`）は:
1. ハンドシェイクの再適用をスキップ（`skipHandshake: true`）
2. サーバーが改めて `ROOM_STATE` で完全な状態を送信
3. クライアントの `Decoder` が状態を上書き復元
4. バッファリングされていたメッセージ（`enqueuedMessages`）を順次送信

## 7. パフォーマンス考慮事項

### 状態サイズ

| 要素 | 影響 | 推奨 |
|------|------|------|
| Schema プロパティ数 | エンコード時間・パッチサイズに比例 | 頻繁に変化するデータのみ state に含める |
| ネストの深さ | Encoder の走査コストが増加 | 浅いフラット構造を推奨 |
| MapSchema/ArraySchema のサイズ | 要素追加・削除のたびにデルタが発生 | 大量の要素は分割を検討 |
| 初期同期サイズ | 新規接続時に fullState が送信される | 大きな state は接続遅延の原因 |

### パッチ頻度

| 設定 | デフォルト | 影響 |
|------|----------|------|
| `patchRate` | 50ms (20fps) | 低すぎると帯域消費増大、高すぎると同期遅延 |
| `setSimulationInterval` | 16.66ms (60fps) | ゲームロジックの更新頻度 |

`patchRate` と `simulationInterval` の分離により、ゲームロジックは高頻度で実行しつつ、ネットワーク送信は低頻度に抑えることが可能。

### フィルタリング（StateView）のコスト

- フィルタなし: パッチは一度だけエンコードされ、全クライアントに同一バイト列を送信
- フィルタあり: クライアントごと（正確には StateView ごと）に `encodeView` が実行される
- 同一 StateView を共有するクライアントには `encodedViews` Map によるキャッシュが効く
- フィルタの有無は `encoder.context.hasFilters` で自動判定

### メッセージレート制限

サーバー側で `maxMessagesPerSecond` プロパティ（デフォルト `Infinity`）が設定可能。超過したクライアントは即座に切断される。

```typescript
if (++client._numMessagesLastSecond > this.maxMessagesPerSecond) {
    this.#_forciblyCloseClient(client, CloseCode.WITH_ERROR);
}
```

### バッファ管理

- サーバー・クライアント双方で `@colyseus/msgpackr` の `Packr` バッファを共有・再利用
- `RESERVE_START_SPACE = 2048` バイトのオフセットで msgpack エンコード位置を制御
- バッファサイズが不足した場合は動的にリサイズ（`packr.useBuffer`）

### 変更なしの最適化

- `encoder.hasChanges === false` の場合、`applyPatches` は即座に `false` を返す
- クライアントが 0 の場合でも変更がある場合は `needFullEncode = true` をセットし、次回の fullState エンコード時に反映
- クライアント 0 + 変更ありの場合は `discardChanges()` で変更をリセット（誰も受け取らないため）

### afterNextPatch キュー

`broadcast` / `broadcastBytes` に `{ afterNextPatch: true }` オプションを渡すと、次のパッチサイクル完了後にメッセージが送信される。状態変更とそれに関連するメッセージの順序保証に使用される。
