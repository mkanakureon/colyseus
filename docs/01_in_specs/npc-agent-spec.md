# AI Agent NPC システム仕様書

**プロジェクト**: kaedevn  
**機能名**: AI Agent NPC（事前生成型 × プレイヤー記憶ナビゲーション）  
**対象プラットフォーム**: ブラウザ版（Web）優先  
**対象フェーズ**: Phase 2（RPG対応）先行実装  
**ステータス**: 仕様策定中（フィーチャーフリーズ解除後に実装予定）  
**作成日**: 2026-03-14

---

## 1. 概要

### 1.1 目的

kaedevnエディタ上でNPCの会話プールを大量に事前生成し、プレイ時はプレイヤーごとの記憶DBを参照して最適な会話を選択・再生する。ブラウザ版を主戦場としてSkyrim的な「世界が呼吸している」体験を実現する。

### 1.2 設計思想

- **生成はエディタ時、選択はランタイム**：重いAPI処理（生成）はエディタ時に完結。プレイ時は軽量な選択処理のみ
- **記憶はナビゲーションに使う**：プレイヤーごとの記憶は「新しい台詞を生成する」のではなく「どの事前生成会話を選ぶか」の判断に使う
- **プール型会話設計**：1NPCあたり大量の会話を用意し、記憶・状態に応じて最適な1本を選択
- **クリエイター制御**：AI生成後に必ず人間がレビュー・承認するワークフロー
- **段階的スケールアップ**：試作は少量で仕組みを検証し、品質確認後に会話数を増やす

### 1.3 Skyrim的体験の実現方法

| 要素 | Skyrim（ランタイム） | kaedevn（事前生成＋記憶選択） |
|------|-------------------|------------------------------|
| 内部状態 | リアルタイム変化 | プレイヤー記憶DBで管理 |
| コンテキスト注入 | ゲーム状態を動的参照 | 記憶・軸条件で会話を選択 |
| 応答生成 | 毎回LLMコール | エディタ時に一括生成済み |
| API負荷 | 毎会話コスト発生 | 選択のみ（追加生成コストなし）|

## 2. 会話プール設計

### 2.1 プール構造

1NPCの会話を3層に分けて管理する。

```typescript
interface NPCConversationPool {
  npcId: string;

  // 日常会話プール（何度でも使い回す・ランダムor記憶選択）
  dailyPool: Conversation[];             // 正式版: 20〜25本

  // 状態連動会話（条件一致時に優先選択）
  contextual: ContextualConversation[];  // 正式版: 10〜15本

  // 特別会話（一度きり・アンロック制）
  special: SpecialConversation[];        // 正式版: 5〜10本

  totalChars: number;                    // 管理用文字数カウント
}
```

### 2.2 会話数の段階設計

試作で仕組みを検証し、品質確認後に増やす。アーキテクチャは変わらない。

| 段階 | NPC数 | 1NPCあたり会話数 | 文字数目安 | 目的 |
|------|-------|----------------|-----------|------|
| **試作** | 2〜3体 | 5〜8本 | 1,500〜2,000文字 | 仕組みの検証 |
| **α版** | 10体程度 | 15〜20本 | 〜5,000文字 | 品質・選択精度の検証 |
| **正式版** | RPG規模 | 40本前後 | 〜10,000文字 | 体験の豊かさ |

### 2.3 軸ベーストリガー設計

会話プールを「何本作るか」ではなく「どの軸で変化させるか」で設計する。

```
関係値軸         低い → 普通 → 高い → 最高
クエスト軸       未開始 → 進行中 → 完了 → 失敗
世界状態軸       平和 → 緊張 → 戦争 → 復興
プレイヤー評判軸  無名 → 有名 → 英雄 → 悪名高い
```

| NPC重要度 | 使用する軸数 | 生成数目安（正式版） |
|----------|------------|-------------------|
| 主要NPC（仲間・キーキャラ） | 3〜4軸 | 30〜50本 |
| 中堅NPC（依頼主・商人） | 2軸 | 10〜15本 |
| モブNPC（通行人・兵士） | 1軸 | 3〜5本 |

## 3. データモデル

### 3.1 NPCDefinition

```typescript
interface NPCDefinition {
  id: string;                    // "npc_innkeeper_01"
  name: string;                  // "マルタ"
  role: string;                  // "宿屋の女将"
  age?: string;                  // "50代"
  appearance?: string;           // 外見の簡単な説明（LLMプロンプトに使用）
  personality: string[];         // ["陽気", "お節介", "秘密主義"]
  background: string;            // キャラクター背景（自由記述）
  initialMemory: string[];       // ["主人公に借りがある", "最近盗難被害にあった"]
  speechStyle: string;           // "関西弁気味のおばちゃん口調"
  pool: NPCConversationPool;
  createdAt: string;
  updatedAt: string;
}
```

### 3.2 Conversation（プール内の1会話）

```typescript
interface Conversation {
  id: string;                    // "conv_daily_001"
  poolType: "daily" | "contextual" | "special";
  label: string;                 // エディタ表示用 "天気の話"

  // 選択条件（contextual / special のみ）
  condition?: ConversationCondition;

  // 会話本体（KNF互換ノード列）
  nodes: KNFConversationNode[];

  approvedAt?: string;
  generatedAt: string;
  modelUsed: string;
}

interface ConversationCondition {
  relationMin?: number;          // 関係値下限
  relationMax?: number;          // 関係値上限
  questFlags?: string[];         // 必要クエストフラグ
  worldState?: string;           // "peace" | "war" etc.
  playerReputation?: string;     // "hero" | "villain" etc.
  once?: boolean;                // trueなら一度再生したら除外
}
```

### 3.3 NPCMemory（プレイヤーごとに保持）

記憶は「生成のインプット」ではなく「会話選択のナビゲーター」として機能する。

```typescript
interface NPCMemory {
  npcId: string;
  playerId: string;

  // 選択軸
  relationScore: number;           // -100〜100
  questFlags: string[];            // 進行中・完了クエスト
  worldStateSnapshot: string;      // 最終会話時の世界状態

  // 再生管理
  playedConversationIds: string[]; // 再生済みID（重複防止）
  lastPlayedAt: string;

  // 記憶ログ（5回ごとに自動圧縮）
  episodes: Episode[];
  lastConversationSummary: string;
}

interface Episode {
  summary: string;               // "プレイヤーが盗賊から村を救った"
  impact: "positive" | "negative" | "neutral";
  timestamp: string;
}
```

### 3.4 KNFConversationNode（既存互換）

```typescript
interface KNFConversationNode {
  id: string;
  speaker: string;
  text: string;
  emotion?: string;              // "happy" | "sad" | "angry" | "neutral" | "surprised"
  choices?: Choice[];
  flags?: FlagOperation[];
}

interface Choice {
  label: string;
  next: string;                  // "node_end" で終了
}

interface FlagOperation {
  flag: string;
  value: boolean | number | string;
}
```

## 4. ランタイム選択ロジック

```
[プレイヤーがNPCに話しかける]
        ↓
NPCMemoryを取得（relationScore・questFlags・playedIds）
        ↓
① specialプールをチェック
   条件一致 + 未再生 → そちらを優先選択
        ↓（なければ）
② contextualプールをチェック
   条件一致のもの → 関係値が最も近いものを選択
        ↓（なければ）
③ dailyプールから選択
   未再生を優先 → すべて再生済みならランダム
        ↓
選択した会話を再生
        ↓
NPCMemoryを更新（playedIds追加・summary更新）
```

選択はルールベースで実装（LLMコール不要）。軽量・低レイテンシ。

## 発見体験設計（サイレントリリース方針）

### 概要

AI機能をリリース時にアピールしない。プレイヤーが遊ぶうちに自然に気づく体験を設計する。アピールはネタバレであり、没入感を損なう。プレイヤー自身の発見がそのまま口コミになる。

```
アピールした場合
  「AIで動くNPCがいます！」
  → プレイヤーが「テストしてやろう」モードになる
  → 没入感ゼロ・体験が壊れる

黙ってリリースした場合
  プレイヤーが普通に遊ぶ
  → 自然に気づく
  → 自分で発見した感動になる
  → SNSで広める
```

### 発見の段階設計

プレイヤーが「気づく」までの体験を段階的に設計する。

```
1週目：「なんか台詞が多いな」（気づきの萌芽）
2週目：「あれ、前に話したこと覚えてる？」（確信の手前）
1ヶ月後：「この世界、生きてる」（完全な没入）

SNSに投稿
  「このゲームのNPC、ちゃんと世界の動き知ってるんだけど…」
  → プレイヤーが宣伝してくれる
```

### 「確信の瞬間」を意図的に設計する

普通の会話が続く中で、一言だけ「覚えている」ことを示す。

```
NPC：「そういえばあなた、先週ここに来た時に
       言ってたこと…あれ、本当だったね。」

プレイヤー：（先週の会話を覚えてる…？）
```

この一言のために、それまでの普通の会話が全部伏線になる。確信の瞬間は自然に生まれるよう設計し、クリエイターが意図的に仕込む。

### サイレントリリースの設計原則

| 原則 | 悪い例 | 良い例 |
|------|-------|-------|
| 馴染ませる | 最初から全NPC全機能フル稼働 | 最初は普通のNPCに見せる |
| 段階的に深める | 初日から記憶言及 | 数回会話後にさりげなく言及 |
| 押しつけない | 毎回ニュースの話をする | 自然な会話の中にさりげなく混ぜる |
| 気づかせる | 「私はAIです」 | プレイヤーが自分で結論を出す |

### リリース後の情報開示タイミング

```
リリース時          何も言わない

プレイヤーの発見がSNSで広まる

開発者（mk18）が後から認める
  「そうです、NPCは毎日世界の動きを知っています。」
  「記憶しています。」

→ 後出しの方が強い
→ プレイヤーの発見を肯定する形になる
→ 話題が二度盛り上がる
```

### kaedevnのマーケティングへの影響

この体験設計はkaedevnプラットフォームの訴求にも使える。

```
クリエイターへの訴求：
  「黙って入れるだけで、プレイヤーが勝手に気づいて広める機能」

プレイヤーへの訴求：
  後から種明かしをする形で
  「kaedevnのゲームには、こういう仕組みがある」
```

## 12. リリース後運用設計（Switch以外の全プラットフォーム対応）

### 12.1 ライブ追加の仕組み

会話データをアプリ本体に埋め込まず、APIでダウンロードする設計にすることで、ブラウザ・スマホ・PCの全プラットフォームでリリース後も会話をライブ追加できる。Switchは通信環境があれば対応可能。

```
従来のゲーム（パッケージ・アプリ本体埋め込み）
  会話追加 → ビルド → ストア審査（1〜7日）
           → ユーザーがアップデート → 反映

kaedevn方式（APIダウンロード設計）
  エディタで会話生成 → 承認 → サーバーDBに追加
  → 起動時 or 話しかける直前にAPIダウンロード
  → 全プレイヤーに自動適用・審査不要
```

### 12.2 アーキテクチャ

```
[kaedevnサーバー]
  会話プールDB
      ↓ API
  ブラウザ / スマホ（iOS・Android） / PC
  → 起動時にダウンロード
  → ローカルキャッシュに保存
  → キャッシュがあればオフラインでも動作

  Switch
  → 通信ありの場合は同様にダウンロード可
  → 通信なしの場合はビルド時埋め込みにフォールバック
```

### 12.3 プラットフォーム別対応表

| プラットフォーム | 会話ライブ追加 | 方法 |
|----------------|-------------|------|
| ブラウザ | ◎ | サーバーから直接取得 |
| スマホ（iOS/Android） | ◎ | 起動時APIダウンロード |
| PC（Steam等） | ◎ | 起動時APIダウンロード |
| Switch（通信あり） | ○ | 起動時APIダウンロード |
| Switch（通信なし） | △ | ビルド時埋め込みにフォールバック |

### 12.4 運用シナリオ例

**通常運用（ゲームを育てる）**
```
リリース時    NPC 10体 × 8本
1ヶ月後      反応の良いNPCに10本追加
3ヶ月後      季節・イベント会話を追加
             → プレイヤーが定期的に戻ってくる理由になる
```

**DLC前の伏線仕込み**
```
DLC発売2週間前
  関連NPCに意味深な会話を追加
  → 「なんか最近マルタの様子がおかしい」
  → プレイヤーが気づき始めSNSで話題になる
  → DLC発売
```

**プレイヤー反応への対応**
```
「このキャラもっと喋ってほしい」という声
  → エディタで追加生成 → 即反映
  → 反応速度がゲームへの信頼感を生む
```

### 12.5 会話追加のルール設計

追加会話がゲームバランスを壊さないよう、クリエイターが管理すべき原則：

| ルール | 理由 |
|--------|------|
| 既存ストーリーの矛盾を作らない | 後から追加した会話が前の伏線と衝突しないよう注意 |
| 追加はdailyプール優先 | specialやcontextualの追加は条件設計が複雑になりやすい |
| 承認フローを必ず通す | ライブ追加でも品質担保のレビューは省略しない |

### 12.6 新しいゲーム運用モデルとしての位置づけ

```
ゲームをSNSのように運用できる

パッケージゲーム：リリースで完結
ブラウザ版NPC  ：継続的にコンテンツが増える
                 プレイヤーとの関係が続く
                 話題が定期的に生まれる
```

これはkaedevnが「ゲームをSNSのように運用できるプラットフォーム」として対外的に訴求できるポイント。Switch以外の全プラットフォームで成立する。
