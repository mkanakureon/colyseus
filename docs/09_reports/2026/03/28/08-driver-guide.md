## 1. Driver インターフェースの解説

### MatchMakerDriver とは

`MatchMakerDriver` は、Colyseus のマッチメイキングシステムがルームキャッシュ（ルーム一覧）を永続化・検索するためのインターフェースである。定義は `packages/core/src/matchmaker/driver.ts` にある。

すべての Driver はこのインターフェースを実装し、ルームの CRUD 操作を提供する。

### インターフェース定義

```typescript
export interface MatchMakerDriver {
  has(roomId: string): Promise<boolean> | boolean;
  query(conditions, sortOptions?): Promise<IRoomCache[]> | IRoomCache[];
  cleanup?(processId: string): Promise<void>;
  findOne(conditions, sortOptions?): Promise<IRoomCache>;
  remove(roomId: string): Promise<boolean> | boolean;
  update(room, operations): Promise<boolean> | boolean;
  persist(room, create?): Promise<boolean> | boolean;
  clear(): void;
  boot?(): Promise<void>;
  shutdown(): void;
}
```

### 各メソッドの役割

| メソッド | 説明 | 必須 |
|---------|------|------|
| `has(roomId)` | 指定 roomId のルームがキャッシュに存在するか確認 | Yes |
| `query(conditions, sortOptions?)` | 条件に合致するルーム一覧を取得 | Yes |
| `findOne(conditions, sortOptions?)` | 条件に合致する最初のルームを取得 | Yes |
| `persist(room, create?)` | ルームをキャッシュに保存。`create=true` で新規作成 | Yes |
| `update(room, operations)` | `$set`（上書き）と `$inc`（増減）の操作でルームを更新 | Yes |
| `remove(roomId)` | ルームをキャッシュから削除 | Yes |
| `cleanup(processId)` | 指定プロセス ID の全ルームを一括削除（スケーリング時のクリーンアップ） | No |
| `boot()` | Driver の初期化処理（テーブル作成など） | No |
| `shutdown()` | 接続の切断・リソース解放 | Yes |
| `clear()` | キャッシュの全消去（テスト用） | Yes |

### IRoomCache の構造

```typescript
export interface IRoomCache<Metadata = any> {
  name: string;          // ルーム名
  roomId: string;        // ルーム固有 ID
  processId: string;     // ルームが動作しているプロセス ID
  clients: number;       // 接続クライアント数
  maxClients: number;    // 最大クライアント数
  locked?: boolean;      // ロック状態（join 拒否）
  private?: boolean;     // プライベートルーム
  publicAddress?: string; // サーバーの公開アドレス
  unlisted?: boolean;    // ロビーリストに非表示
  metadata?: Metadata;   // カスタムメタデータ
  createdAt?: Date;      // 作成日時
}
```

### update の操作構文

MongoDB 風の操作構文を採用している。

```typescript
// $set: フィールドの値を上書き
await driver.update(room, { $set: { locked: true } });

// $inc: 数値フィールドを増減
await driver.update(room, { $inc: { clients: 1 } });

// 組み合わせ
await driver.update(room, {
  $set: { locked: true },
  $inc: { clients: -1 }
});
```

### LocalDriver の実装（リファレンス実装）

`LocalDriver`（`packages/core/src/matchmaker/LocalDriver/LocalDriver.ts`）はインメモリ配列でルームキャッシュを管理するリファレンス実装である。

- ストレージ: `IRoomCache[]` 配列
- クエリ: `Query` クラスによるフィルタリング + ソート（`Array.filter` / `Array.sort`）
- 同期処理: すべてのメソッドが同期的に動作（Promise 不要）
- 外部依存: なし

これがデフォルトの Driver であり、単一プロセスでの開発・運用に適している。

| 項目 | LocalDriver | RedisDriver | MongooseDriver | PostgresDriver (Drizzle) |
|------|------------|-------------|----------------|--------------------------|
| **パッケージ名** | `@colyseus/core` (内蔵) | `@colyseus/redis-driver` | `@colyseus/mongoose-driver` | `@colyseus/drizzle-driver` |
| **バージョン** | 0.17.x | 0.17.7 | 0.17.6 | 0.17.7 |
| **バックエンド** | インメモリ配列 | Redis Hash | MongoDB (Mongoose) | PostgreSQL (Drizzle ORM) |
| **依存ライブラリ** | なし | ioredis ^5.3.2 | mongoose ^8.4.4 | drizzle-orm ^0.38.4, postgres ^3.4.7 |
| **マルチプロセス対応** | 不可 | 可能 | 可能 | 可能 |
| **永続化** | なし（プロセス終了で消失） | Redis の永続化設定に依存 | MongoDB に永続化 | PostgreSQL に永続化 |
| **クラスタ対応** | - | Redis Cluster 対応 | Replica Set 対応 | - |
| **boot() メソッド** | なし | なし | なし | あり（テーブル自動作成） |
| **クエリ方式** | 配列フィルタ | Redis HGETALL + JS フィルタ | MongoDB ネイティブクエリ | SQL WHERE 句 |
| **メタデータクエリ** | JS オブジェクト走査 | JSON パース後フィルタ | MongoDB Mixed type | JSONB 演算子 (`->>`) |
| **レースコンディション対策** | 不要（シングルプロセス） | JS 側で処理 | MongoDB アトミック操作 | SQL アトミック演算（`$inc` 時に `locked` を SQL で計算） |
| **セットアップ難易度** | なし | 低（Redis サーバーが必要） | 中（MongoDB が必要） | 中（PostgreSQL + テーブル作成） |

## 3. 各 Driver のセットアップ方法

### 3.1 LocalDriver（デフォルト）

設定不要。Colyseus サーバーのデフォルト Driver として組み込み済み。

```typescript
import { Server } from "colyseus";

const server = new Server({
  // driver を指定しなければ LocalDriver が使われる
});
```

**適用場面:** 開発環境、単一プロセスでの運用、プロトタイピング。

---

### 3.2 RedisDriver

```bash
npm install @colyseus/redis-driver
```

```typescript
import { Server } from "colyseus";
import { RedisDriver } from "@colyseus/redis-driver";

// 基本的な接続
const server = new Server({
  driver: new RedisDriver(),
  // デフォルト: localhost:6379
});

// 接続オプション指定
const server = new Server({
  driver: new RedisDriver({
    host: "redis.example.com",
    port: 6379,
    password: "your-password",
  }),
});

// 既存の Redis/Cluster インスタンスを渡す
import { Redis } from "ioredis";
const redis = new Redis("redis://redis.example.com:6379");
const server = new Server({
  driver: new RedisDriver(redis),
});

// Redis Cluster
import { Cluster } from "ioredis";
const server = new Server({
  driver: new RedisDriver([
    { host: "node1.example.com", port: 6379 },
    { host: "node2.example.com", port: 6379 },
    { host: "node3.example.com", port: 6379 },
  ], { /* ClusterOptions */ }),
});
```

**内部動作:**
- `roomcaches` という Redis Hash にルーム情報を JSON 文字列で保存
- `HGETALL` で全ルームを取得し、JavaScript 側でフィルタリング
- 同一 tick 内の複数リクエストは `_concurrentRoomCacheRequest` で共有されるため、Redis への読み取りが最適化される
- cleanup は 500 件ずつのバッチ削除

**内部動作:**
- `RoomCache` という Mongoose モデルを使用
- `name + locked` と `roomId` にインデックスが作成される
- MongoDB のネイティブクエリ（`find`, `findOne`, `updateOne`）を使用するため、大量ルームでも効率的
- `$set` / `$inc` はそのまま MongoDB の update operators として渡される

### 3.4 PostgresDriver (Drizzle)

```bash
npm install @colyseus/drizzle-driver
```

```typescript
import { Server } from "colyseus";
import { PostgresDriver } from "@colyseus/drizzle-driver";

// デフォルト接続（環境変数 DATABASE_URL または localhost）
const server = new Server({
  driver: new PostgresDriver(),
});
// boot() でテーブルが自動作成される（roomcaches_v1）

// 既存の Drizzle DB インスタンスを渡す
import postgres from "postgres";
import { drizzle } from "drizzle-orm/postgres-js";

const sql = postgres("postgresql://user:pass@host:5432/db");
const db = drizzle(sql);

const server = new Server({
  driver: new PostgresDriver({ db }),
});
// 注意: 外部 DB インスタンスを渡した場合、boot() は使用不可
// テーブルは自分で作成する必要がある

// カスタムスキーマ
import { roomcaches } from "@colyseus/drizzle-driver";
import { pgTable, varchar, integer, boolean, timestamp, jsonb } from "drizzle-orm/pg-core";

const customRoomcaches = pgTable("my_roomcaches", {
  roomId: varchar({ length: 9 }).primaryKey(),
  processId: varchar({ length: 9 }),
  name: varchar({ length: 64 }).notNull(),
  clients: integer().notNull(),
  maxClients: integer().notNull(),
  locked: boolean(),
  private: boolean(),
  metadata: jsonb(),
  publicAddress: varchar({ length: 255 }),
  createdAt: timestamp().notNull().defaultNow(),
  unlisted: boolean(),
});

const server = new Server({
  driver: new PostgresDriver({ schema: customRoomcaches }),
});
```

**内部動作:**
- `roomcaches_v1` テーブルを PostgreSQL に作成
- メタデータは JSONB 型で保存。クエリ時に `metadata->>'fieldName'` で条件指定
- `$inc` 操作時に `locked` フラグを SQL で原子的に計算し、レースコンディションを防止
- `maxClients` が `Infinity` の場合、PostgreSQL の最大整数値（2,147,483,647）に変換

## 4. Presence バックエンドの解説

### Presence とは

`Presence` は Colyseus プロセス間の通信・データ共有を担うインターフェースである。マッチメイキング時のプロセス間調整や、Pub/Sub によるメッセージングに使用される。

定義は `packages/core/src/presence/Presence.ts` にある。

### Presence インターフェースの主要機能

| カテゴリ | メソッド | 説明 |
|---------|---------|------|
| **Pub/Sub** | `subscribe`, `unsubscribe`, `publish`, `channels` | トピックベースのメッセージング |
| **Key-Value** | `set`, `get`, `setex`, `expire`, `del`, `exists` | 単純な KV ストア |
| **Set** | `sadd`, `smembers`, `sismember`, `srem`, `scard`, `sinter` | 集合操作 |
| **Hash** | `hset`, `hget`, `hgetall`, `hdel`, `hlen`, `hincrby` | ハッシュマップ操作 |
| **Counter** | `incr`, `decr` | アトミックカウンター |
| **List** | `rpush`, `lpush`, `rpop`, `lpop`, `brpop`, `llen` | リスト操作 |

### 4.1 LocalPresence（デフォルト）

`packages/core/src/presence/LocalPresence.ts` に実装。

- **ストレージ:** Node.js の `EventEmitter`（Pub/Sub）+ オブジェクト（KV/Set/Hash）
- **マルチプロセス:** 不可。単一プロセス内でのみ動作
- **DevMode:** `isDevMode` の場合、shutdown 時にキャッシュをファイルに保存し、再起動時に復元
- **TTL:** `setTimeout` でキーの有効期限を実装
- **brpop:** ポーリングで実装（タイムアウト時間の間、125ms 間隔でリトライ）

```typescript
import { Server } from "colyseus";

const server = new Server({
  // presence を指定しなければ LocalPresence が使われる
});
```

### 4.2 RedisPresence

`packages/presence/redis-presence/src/index.ts` に実装。

- **ストレージ:** Redis（ioredis ^5.3.2）
- **マルチプロセス:** 可能。Redis の Pub/Sub で複数プロセス間通信
- **接続:** Pub 用と Sub 用の2つの Redis クライアントを使用（Redis の制約: SUBSCRIBE 中のクライアントではコマンドを実行できない）
- **クラスタ:** Redis Cluster 対応

```bash
npm install @colyseus/redis-presence
```

```typescript
import { Server } from "colyseus";
import { RedisPresence } from "@colyseus/redis-presence";

// デフォルト接続
const server = new Server({
  presence: new RedisPresence(),
});

// 接続オプション指定
const server = new Server({
  presence: new RedisPresence({
    host: "redis.example.com",
    port: 6379,
    password: "your-password",
  }),
});

// 既存の Redis インスタンスを渡す（sub 用に自動で duplicate される）
import { Redis } from "ioredis";
const redis = new Redis("redis://redis.example.com:6379");
const server = new Server({
  presence: new RedisPresence(redis),
});

// Redis Cluster
const server = new Server({
  presence: new RedisPresence([
    { host: "node1.example.com", port: 6379 },
    { host: "node2.example.com", port: 6379 },
  ]),
});
```

### ScopedPresence

`Presence.ts` には `createScopedPresence()` ユーティリティも定義されている。これはルームごとのサブスクリプションを追跡し、ルーム dispose 時に自動で unsubscribe するラッパーである。ルーム内で `this.presence` として使用される。

## 5. スケーリング時の Driver + Presence の組み合わせ推奨

### 組み合わせ表

| 構成 | Driver | Presence | 適用場面 |
|------|--------|----------|---------|
| **開発環境** | LocalDriver | LocalPresence | ローカル開発、プロトタイピング |
| **単一サーバー本番** | LocalDriver | LocalPresence | 小規模運用（1プロセス） |
| **マルチプロセス（推奨）** | RedisDriver | RedisPresence | 複数ワーカー、PM2/Docker |
| **マルチプロセス + 永続化** | PostgresDriver | RedisPresence | ルーム情報のクラッシュ復旧が必要な場合 |
| **マルチプロセス + MongoDB** | MongooseDriver | RedisPresence | 既に MongoDB を使用している場合 |

### 重要なルール

1. **マルチプロセスでは Driver と Presence の両方を外部バックエンドにする必要がある。** LocalDriver + RedisPresence や RedisDriver + LocalPresence という組み合わせは不整合を起こす。

2. **Presence は常に Redis 一択。** Presence のマルチプロセス対応は `@colyseus/redis-presence` のみ提供されている。PostgreSQL や MongoDB の Presence 実装は存在しない。

3. **Driver と Presence で同じ Redis インスタンスを共有できる。**

```typescript
import { Redis } from "ioredis";
import { Server } from "colyseus";
import { RedisDriver } from "@colyseus/redis-driver";
import { RedisPresence } from "@colyseus/redis-presence";

const redis = new Redis("redis://redis.example.com:6379");

const server = new Server({
  driver: new RedisDriver(redis),
  presence: new RedisPresence(redis),
  // RedisPresence は内部で redis.duplicate() を呼ぶため、
  // 同一インスタンスを渡しても問題ない
});
```

4. **PostgresDriver は boot() が必要。** サーバー起動時にテーブルが自動作成されるが、外部 DB インスタンスを渡した場合は自分でテーブルを作成する必要がある。

## 6. 選定フローチャート

```
プロセス数は？
|
+-- 1プロセス（開発 or 小規模本番）
|   |
|   +-- DB に永続化が必要？
|       |
|       +-- No  --> LocalDriver + LocalPresence
|       +-- Yes --> PostgresDriver + LocalPresence
|                   (※ スケールアウト予定なら最初から Redis 構成を推奨)
|
+-- 複数プロセス（PM2 cluster / Docker / K8s）
    |
    +-- 既存のインフラは？
        |
        +-- Redis がある
        |   |
        |   +-- ルーム情報の永続化が必要？
        |       |
        |       +-- No  --> RedisDriver + RedisPresence  [最もシンプル]
        |       +-- Yes --> PostgresDriver + RedisPresence
        |
        +-- MongoDB がある（Redis もある）
        |   |
        |   +-- MongooseDriver + RedisPresence
        |
        +-- PostgreSQL がある（Redis もある）
        |   |
        |   +-- PostgresDriver + RedisPresence
        |
        +-- Redis がない
            |
            +-- Redis を導入する（Presence に必須）
                --> 上記いずれかの組み合わせ
```

### 判断基準まとめ

| 判断ポイント | 推奨 |
|-------------|------|
| 最もシンプルなマルチプロセス構成 | RedisDriver + RedisPresence |
| 既に PostgreSQL を使っている | PostgresDriver + RedisPresence |
| 既に MongoDB を使っている | MongooseDriver + RedisPresence |
| レースコンディション耐性が最も重要 | PostgresDriver（SQL アトミック演算） |
| 開発環境 | LocalDriver + LocalPresence（デフォルト） |
| Redis Cluster でのスケーリング | RedisDriver + RedisPresence（両方 Cluster 対応） |
