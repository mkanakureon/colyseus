## 1. Colyseus メッセージング基盤の仕組み

### 1.1 プロトコルコード（トランスポート層）

Colyseus は WebSocket 上で独自のバイナリプロトコルを使用する。`@colyseus/shared-types` の `Protocol` で定義される：

```typescript
// packages/shared-types/src/Protocol.ts
export const Protocol = {
  JOIN_ROOM: 10,        // Room 参加確認
  ERROR: 11,            // エラー通知
  LEAVE_ROOM: 12,       // Room 退出
  ROOM_DATA: 13,        // アプリケーションメッセージ（msgpack）
  ROOM_STATE: 14,       // 状態全体送信
  ROOM_STATE_PATCH: 15, // 状態差分パッチ
  ROOM_DATA_BYTES: 17,  // バイナリメッセージ
  PING: 18,             // Ping/Pong
} as const;
```

アプリケーション層のメッセージ（本仕様書で定義するもの）は、すべて `ROOM_DATA (13)` のペイロードとして送受信される。

### 1.2 サーバー側 API（packages/core/src/Room.ts）

| メソッド | 用途 | 送信先 |
|---------|------|--------|
| `this.broadcast(type, message, options?)` | 全クライアントに送信 | Room 内全員（`except` で除外可） |
| `client.send(type, message)` | 特定クライアントに送信 | 1クライアント |
| `this.onMessage(type, callback)` | クライアントからのメッセージ受信ハンドラ登録 | — |
| `this.onMessage("*", callback)` | 全メッセージのフォールバックハンドラ | — |

サーバーは `messages` プロパティでも宣言的にハンドラを登録できる：

```typescript
class MyRoom extends Room {
  messages = {
    move: (client, message) => { /* ... */ },
    chat: validate(z.object({ text: z.string() }), (client, message) => { /* ... */ }),
  };
}
```

### 1.3 クライアント側 API（packages/sdk/src/Room.ts）

| メソッド | 用途 |
|---------|------|
| `room.send(type, payload?)` | サーバーへメッセージ送信（msgpack シリアライズ） |
| `room.sendBytes(type, bytes)` | サーバーへバイナリ送信 |
| `room.sendUnreliable(type, message?)` | 非信頼送信（UDP/WebTransport） |
| `room.onMessage(type, callback)` | サーバーからのメッセージ受信ハンドラ登録 |
| `room.onMessage("*", callback)` | 全メッセージのフォールバックハンドラ |

接続が切断中の場合、`send()` はメッセージをバッファリングし、再接続後に自動送信する（最大 `maxEnqueuedMessages` = 10 件）。

### 1.4 状態同期（@colyseus/schema）

`ROOM_DATA` とは別に、Room の `state` プロパティ（`@colyseus/schema`）が自動的にデルタ圧縮で同期される：
- 初回: `ROOM_STATE (14)` で全状態送信
- 以降: `ROOM_STATE_PATCH (15)` で差分パッチ（デフォルト 50ms / 20fps）

本仕様書の「メッセージ」は `ROOM_DATA` で送受信されるアプリケーション層メッセージを指す。`state` の自動同期は別チャネルである。

## 7. エラーメッセージ定義（全 Room 共通）

すべての Room で `"error"` メッセージ型を使用してアプリケーションレベルのエラーを通知する。

```typescript
/** アプリケーションエラー（client.send） */
interface AppError {
  code: AppErrorCode;
  message: string;        // 人間が読めるエラーメッセージ（日本語）
  details?: Record<string, unknown>;
}

type AppErrorCode =
  // 認証 (1xxx)
  | "AUTH_FAILED"            // 1001: JWT 検証失敗
  | "AUTH_EXPIRED"           // 1002: トークン期限切れ
  | "AUTH_SUSPENDED"         // 1003: アカウント停止
  | "AUTH_GUEST_RESTRICTED"  // 1004: ゲストユーザーの機能制限

  // WorldRoom (2xxx)
  | "ZONE_NOT_FOUND"        // 2001: 移動先ゾーンが存在しない
  | "ZONE_FULL"             // 2002: 移動先ゾーンが満員
  | "ZONE_NO_EXIT"          // 2003: 指定方向に出口がない
  | "NPC_NOT_FOUND"         // 2004: 対象 NPC が存在しない
  | "INTERACT_TOO_FAR"      // 2005: NPC が遠すぎる

  // ChatRoom (3xxx)
  | "CHAT_RATE_LIMIT"       // 3001: メッセージ連投制限
  | "CHAT_TOO_LONG"         // 3002: メッセージが長すぎる（500文字超）
  | "CHAT_EMPTY"            // 3003: 空メッセージ
  | "CHAT_TARGET_NOT_FOUND" // 3004: ウィスパー先が見つからない
  | "CHAT_INVALID_CHANNEL"  // 3005: 不正なチャンネル指定

  // BattleRoom (4xxx)
  | "BATTLE_NOT_YOUR_TURN"  // 4001: 自分のターンではない
  | "BATTLE_INVALID_ACTION" // 4002: 無効なアクション種別
  | "BATTLE_INVALID_TARGET" // 4003: 無効な対象
  | "BATTLE_TARGET_DEAD"    // 4004: 対象が既に死亡
  | "BATTLE_NO_ITEM"        // 4005: アイテムを所持していない
  | "BATTLE_NO_MP"          // 4006: MP 不足
  | "BATTLE_ALREADY_ACTED"  // 4007: 既にアクション済み
  | "BATTLE_CANNOT_FLEE"    // 4008: 逃走不可

  // TradeRoom (5xxx)
  | "TRADE_INVALID_ITEM"    // 5001: 所持していないアイテム
  | "TRADE_INSUFFICIENT"    // 5002: 数量不足
  | "TRADE_ALREADY_ACCEPTED"// 5003: 既に承認済み
  | "TRADE_NOT_FOUND"       // 5004: 取引が存在しない
  | "TRADE_GOLD_INSUFFICIENT" // 5005: ゴールド不足

  // 汎用 (9xxx)
  | "INVALID_PAYLOAD"       // 9001: 不正なメッセージ形式

  ;

// 使用例:
// client.send("error", { code: "ZONE_NO_EXIT", message: "その方向には出口がありません。" })
```

### Colyseus トランスポートレベルのエラーとの使い分け

| レベル | 仕組み | 用途 |
|--------|--------|------|
| トランスポート | `ErrorCode.AUTH_FAILED (525)` + WebSocket close | 認証失敗、Room 参加拒否 |
| トランスポート | `client.error(code, message)` → `Protocol.ERROR (11)` | 致命的エラー（接続切断を伴う） |
| アプリケーション | `client.send("error", AppError)` → `Protocol.ROOM_DATA (13)` | 操作エラー（接続維持） |

## 8. インラインタグ付きメッセージの扱い

### 8.1 概要

kaedevn エンジンのインラインディレクティブ（`[e:][p:][cam:]` 等）は、NPC 対話・戦闘ログ・システムメッセージの `text` フィールドに埋め込まれる。Colyseus メッセージとしてはプレーンな文字列であり、特別なシリアライズは不要。

### 8.2 タグ一覧

| タグ | 説明 | 例 |
|------|------|-----|
| `[e:名前]` | 表情変更（18種） | `[e:smile]`, `[e:angry]` |
| `[p:名前]` | ポーズ変更（16種） | `[p:wave]`, `[p:fighting]` |
| `[t]` `[w]` `[s]` | リップシンク（talk/whisper/shout） | `[t]こんにちは` |
| `[cam:shake]` | 画面揺れ | `[cam:shake]` |
| `[cam:zoom:倍率]` | ズーム | `[cam:zoom:1.5]` |
| `[cam:flash]` | 白フラッシュ | `[cam:flash]` |
| `[cam:dark]` | 暗転 | `[cam:dark]` |
| `[cam:reset]` | カメラリセット | `[cam:reset]` |
| `[wait:ミリ秒]` | 一時停止 | `[wait:500]` |
| `[click]` | クリック待ち | `[click]` |

### 8.3 タグを含むメッセージの流れ

```
サーバー (BattleRoom)
  │
  │ this.broadcast("action_result", {
  │   log: "[e:angry][cam:shake]エミリの攻撃！[e:sad]スライムに10のダメージ！"
  │ })
  │
  ▼ Protocol.ROOM_DATA (13) + msgpack シリアライズ
  │
WebSocket バイナリフレーム
  │
  ▼ SDK room.onMessage("action_result", callback)
  │
クライアント (kaedevn Engine)
  │ parseInlineText(log) → セグメント分解
  │   [0] { kind: "directive", directive: { type: "expr", value: "angry" } }
  │   [1] { kind: "directive", directive: { type: "cam", value: "shake" } }
  │   [2] { kind: "text", text: "エミリの攻撃！" }
  │   [3] { kind: "directive", directive: { type: "expr", value: "sad" } }
  │   [4] { kind: "text", text: "スライムに10のダメージ！" }
  │
  ▼ executeDirective → 表情変更・カメラ演出 + TextWindow でテキスト表示
```

### 8.4 タグの処理責任

| 処理 | 責任 | 場所 |
|------|------|------|
| タグの生成 | サーバー（NPC 対話、戦闘ログ生成） | Room 実装 |
| タグのパース | クライアント | `parseInlineText()` |
| タグの実行（表情変更・カメラ等） | クライアント | `executeDirective()` |
| タグの除去（ログ保存用） | サーバーまたはクライアント | `stripTags()` |

サーバーはタグを含む文字列をそのまま送信し、クライアントがパース・実行する。サーバーはタグの意味を解釈しない（透過的に転送する）。
